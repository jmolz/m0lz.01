# Feature: v0.3.0 Dogfood Hardening

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Five connected fixes surfaced during the first live dogfood of `blog publish` against `m0lz.00`. Every fix closes a gap that silently produces a broken artifact (missing frontmatter field, wrong PR contents, unseeded pipeline, ambiguous preview URL). None of them are independently shippable — #4 is the recovery path for #1's upstream failure, and #3 is the control-flow patch that would have prevented the operator-visible symptom of #1 in the first place.

## User Story

As an operator running `blog publish` on a project-launch post, I want every pre-publish validation to fail loudly when state is incomplete, and I want a supported recovery path when a mistake slipped past the gates, so that I never hand-edit a generated PR or run SQL against the state DB to unblock a cycle.

## Success Criteria

- [ ] `research init` on a `project-launch` content type refuses to create a post when `project_id` cannot be resolved from either a `--project` flag, a catalog-ID regex match against the prompt, or an explicit config lookup.
- [ ] `generateFrontmatter` emits `companion_repo: <git-origin-url>` for every post where `post.project_id` maps to a path in `config.projects`.
- [ ] Site-PR step refuses to cut a feature branch when local `main` is ahead of `origin/main`, unless the operator passes an explicit override flag that propagates into the plan-file hash.
- [ ] `runPublishStart` seeds `pipeline_steps` for every post entering phase=`publish`, including the case where `completeEvaluation` already advanced the phase — zero-row state is never observable at step-dispatch time.
- [ ] Preview-gate emits three URLs (canonical `/writing/<slug>`, supplementary `/research/<slug>`, companion repo resolution) as structured output that the `/blog` skill can render back to the operator.
- [ ] New `blog draft regenerate-frontmatter <slug>` command re-emits frontmatter in-place on `.blog-agent/drafts/<slug>/index.mdx` from the current post row + config, without touching any site-repo file.
- [ ] All new guards are covered by regression tests registered in both `.claude/commands/review.md` and `.windsurf/workflows/review.md` per the testing rule.
- [ ] No regression in the existing 878-test suite.

## Testing Strategy

### Unit Tests

- **Where:** `tests/{new-file}.test.ts` for each new slice; extensions to existing files where they overlap a logical area.
- **Pattern:** Mirror `tests/research-cli.test.ts` for CLI tests (mkdtemp, file-backed DB, injectable paths), `tests/draft-frontmatter.test.ts` for pure-function tests (makePost/makeConfig helpers, no DB).
- **Minimum:** 1 happy path + 1 edge case + 1 error case per public function.

### Integration Tests

- `tests/publish-seed-on-start.test.ts`: drive `completeEvaluation` → verify `posts.phase === 'publish'` AND `listSteps(db, slug).length === 0` → call `runPublishStart` → verify `listSteps(db, slug).length === 11` AND no duplicate steps inserted on second `runPublishStart` call.
- `tests/publish-origin-divergence.test.ts`: spin up two bare repos in temp dirs (origin + clone), create a divergence by committing to clone without pushing, run `createSitePR`, assert `[AGENT_ERROR] ORIGIN_OUT_OF_SYNC` is thrown. Then re-run with `allowMainAhead: true` and assert it proceeds.

### E2E Validation Steps

Outside the automated suite, the post-merge dogfood cycle is the real E2E:

1. Build: `npm run build`
2. Initialize a fresh test workspace: `mkdir /tmp/blog-e2e && cd /tmp/blog-e2e && node ${M0LZ}/dist/cli/index.js init`
3. Register a project: `echo 'projects:\n  test.01: "/tmp/test-proj"' >> .blogrc.yaml`
4. `blog research init --slug test-launch --mode directed --topic "ship test.01 to production" --project test.01`
5. Verify `post.project_id === 'test.01'` via `blog research show test-launch --json`
6. Proceed through pipeline; at preview-gate, verify the structured output includes canonical + supplementary + companion URL lines.

## Adversarial Review

**Tier**: 2
**Reviewers**: self-critique (context-constrained; full Tier 2 Codex pass to run via `/evaluate` after implementation)
**Refinement cycles**: 1
**Attack framework**: Karpathy four principles + PoetiQ cross-model verification

### Critical (resolved before proceeding)

1. **HIDDEN ASSUMPTION** — `config.projects[id]` shape. The plan assumes values are filesystem paths (confirmed via loader.ts:115-116 and publish/readme.ts:61), not URLs. Companion repo URL resolution therefore requires reading `git remote get-url origin` at frontmatter-gen time — NOT a direct config lookup. Section: Task 4. Resolution: Task 4 explicitly documents git-origin resolution with SSH→HTTPS normalization and best-effort degradation on missing origin.

2. **HIDDEN ASSUMPTION** — phase-advance seeding bug root cause. Initial read was "completeEvaluation advances phase without seeding" implying the fix belongs in evaluate/state.ts. Actual: `initPublish` already seeds unconditionally (phase.ts:100); the bug is that `runPublishStart` skips `initPublish` entirely when phase=`publish`. Section: Task 8. Resolution: Task 8 removes the phase-conditional guard in CLI, avoiding a plumbing change across module boundaries (evaluate doesn't have content_type/config plumbing).

3. **SCOPE DRIFT** — original plan included auto-detect of project_id via `m0lz\.\d+` regex. The project is a reusable blog agent, not m0lz-specific. Generic pattern `/\b([a-z][a-z0-9]*\.\d+)\b/i` matches any catalog-style ID (user.01, project.02, repo.42) without hardcoding identity. Section: Task 1. Resolution: regex broadened; m0lz bias removed.

4. **WEAK SUCCESS CRITERIA** — "preview-gate shows URLs correctly" would have been unverifiable. Rewritten as "PreviewGateResult has three specific keys in JSON output with specific resolution behavior per case." Section: Contract criterion #5. Resolution: criterion now names exact keys and null-vs-value cases.

### Consider (flagged, not acted on)

1. **OVERCOMPLICATION** — the `.frontmatter-regenerated.json` receipt file in Task 10 adds plumbing for an audit trail that may not be read by anything. Could be a simple stderr log line instead. Keeping it because the plan file hash + apply-receipt pattern is load-bearing elsewhere in the codebase (`.claude/rules/plan-file.md`) and this is consistent with that convention. User may prefer to cut it.

2. **SCOPE DRIFT** — Task 9 extends `PreviewGateResult` with three URL fields, but the plan does NOT update the `/blog` skill's preview-gate checkpoint prompt to surface these to the operator. That skill-level change could be a Task 15 or deferred to v0.3.1.

3. **VERIFICATION-LOOP READINESS** — Task 5 says "every `generateFrontmatter` call site" but doesn't enumerate them. An executor would need to `grep -rn generateFrontmatter src/` at task-start time. That's fine, but flag it.

### Reasoning divergence (alternative paths not taken)

1. **Cache companion_repo URL at research-init time** — resolve once and store in `posts.repo_url` instead of resolving at every frontmatter-gen call. Faster, but introduces a staleness risk if the project's origin URL changes mid-cycle. Rejected: origin lookups are cheap (local git), and fresh-read matches the pattern used in origin-guard.ts.

2. **Extend `config.projects` schema to include `repo_url` field** — `projects: { m0lz.01: { path: "../m0lz.01", repo_url: "https://github.com/jmolz/m0lz.01" } }`.
