# Feature: Complete Publication Bundle

> Generated by the migrated `plan-feature` command. Read `AGENTS.md` and referenced files before implementing.

## Overview

Make `content/posts/<slug>/` in the hub site repo the complete durable publication bundle for every enabled publication channel. After initial publish, update publish, or explicit backfill, the post directory must contain the canonical MDX, all required assets, all enabled paste/distribution files, and manifest provenance needed to publish or repost the article on external platforms.

The current pipeline splits the bundle across two places:

- Durable site repo: `content/posts/<slug>/index.mdx`, copied draft `assets/`, and later `distribution/linkedin.md`, `distribution/hackernews.md`, optional `distribution/linkedin-image-prompt.md`, `distribution/manifest.json`, and optional `assets/linkedin-feed.png`.
- Workspace-only: `.blog-agent/social/<slug>/medium-paste.md` and `.blog-agent/social/<slug>/substack-paste.md`.

The current code also preserves image-backed MDX visuals in Medium/Substack paste files, but it does not generate portable images for plain Markdown tables. Tables remain fragile when pasted into Medium or Substack.

This feature creates one operator-facing invariant:

`content/posts/<slug>/` is the durable source for every enabled publication and distribution artifact the operator needs.

Config semantics after this feature:

- `publish.medium` controls whether `medium-paste.md` is generated and persisted.
- `publish.substack` controls whether `substack-paste.md` is generated and persisted.
- `social.distribution_kit.enabled=false` disables LinkedIn/Hacker News/prompt/LinkedIn-image generation, but it does not suppress Medium/Substack paste artifacts when those publish channels are enabled.
- `social.distribution_kit.persist_to_site=false` disables the post-preview direct-push/backfill persistence step, but it does not remove complete-bundle files from the reviewed site PR/update PR. Site PR/update PR is the primary durable bundle path.

## User Story

As a technical author publishing through `blog`, I want each hub-site post directory to contain the canonical article, assets, Medium/Substack paste files, social copy, generated images, and manifest provenance, so that I do not have to hunt through `.blog-agent/` or recreate tables manually when distributing a post.

## Success Criteria

- [ ] `content/posts/<slug>/distribution/medium-paste.md` is persisted for initial publish, update publish, and published-post backfill when `publish.medium` is enabled.
- [ ] `content/posts/<slug>/distribution/substack-paste.md` is persisted for initial publish, update publish, and published-post backfill when `publish.substack` is enabled.
- [ ] `content/posts/<slug>/distribution/manifest.json` records SHA-256 provenance for every enabled text artifact, prompt artifact, generated image artifact, and generated table asset.
- [ ] Markdown pipe tables outside code fences are converted for Medium/Substack paste output into deterministic PNG assets under `content/posts/<slug>/assets/`.
- [ ] Generated table images use stable filenames and do not produce timestamp-only churn or duplicate numbered files on repeat runs.
- [ ] Generated table images are rendered with the existing local `sharp` dependency; no browser automation, remote image APIs, ImageMagick, or new runtime dependencies.
- [ ] Site PR/update PR contains the complete post bundle before preview review, not only after a later direct push.
- [ ] The persist/backfill path validates manifest hashes before copying existing local artifacts to the site repo.
- [ ] Published-post backfill uses hub-site `content/posts/<slug>/index.mdx` as the primary paste source, falling back to draft MDX only when the hub copy is absent and the post is not yet published.
- [ ] Existing Dev.to, platform-image, LinkedIn, Hacker News, update, and publish resume behavior remains intact.
- [ ] All validation commands pass.

## Current State

Initial publish steps are:

```text
verify -> research-page -> site-pr -> preview-gate -> crosspost-devto -> paste-medium -> paste-substack -> companion-repo -> update-frontmatter -> update-readme -> social-text
```

Update publish steps are:

```text
verify -> research-page -> site-update -> preview-gate -> crosspost-devto -> paste-medium -> paste-substack -> update-frontmatter -> social-text
```

`createSitePR` generates the current distribution kit before mutating the site repo, but it does not copy that kit into the PR. The kit is copied later by `social-text` through a direct `main` push after the preview gate.

`generateMediumPaste` and `generateSubstackPaste` run after preview gate and write only into `.blog-agent/social/<slug>/`. They are not part of the site PR, not part of the site repo distribution directory, and not represented in `distribution/manifest.json`.

`createSitePR` copies all draft assets into `content/posts/<slug>/assets/`. Generated table images must not be written into `.blog-agent/drafts/<slug>/assets/` during publish because the draft has already passed evaluation artifact verification. Table images are deterministic derived artifacts and must live under the publication bundle workspace, then be copied into the site post `assets/` directory by the same bundle-copy helper that copies distribution files.

## Strategic Design

### Core Decision

Broaden the distribution kit into a complete publication bundle manifest instead of introducing a parallel manifest.

Rationale:

- The distribution kit already owns durable social artifacts and hash validation.
- `social-text` already persists distribution artifacts to the site repo.
- `generateDistributionKit` already runs before `site-pr`, so it can derive Medium/Substack paste files and table assets before `createSitePR` copies bundle files into the PR.
- A single manifest avoids competing provenance files for the same post bundle.

Generation and copy must remain separate:

- Local generation happens before site mutation and writes under `.blog-agent/social/<slug>/` and `.blog-agent/social/<slug>/assets/`.
- Site copying happens after the site branch exists and `content/posts/<slug>/` has been created.
- Post-preview steps are load/verify-only for bundle artifacts when the manifest is enabled. They must not regenerate paste files or table assets after preview review.
- Step 11 may direct-push only when the merged site manifest/assets are missing or exactly match the local manifest's expected write set. If the site already has a different manifest or different artifact bytes, Step 11 fails and requires a fresh PR/update cycle or explicit backfill.

### Target Artifact Layout

```text
content/posts/<slug>/
  index.mdx
  assets/
    devto-cover.png
    medium-featured.png
    substack-preview.png
    portable-table-<hash>.png
    linkedin-feed.png                # only when image-mode generate|required
    <author-provided assets>
  distribution/
    linkedin.md
    hackernews.md
    medium-paste.md
    substack-paste.md
    linkedin-image-prompt.md         # when image-mode prompt-only|generate|required
    manifest.json
```

### Table Rendering Contract

Only convert Markdown pipe tables outside fenced code blocks.

A detected table must have:

- at least one header row,
- a separator row containing pipes and dashes/colons,
- at least one body row,
- no unmatched code fence state.

Render each detected table to a deterministic SVG and then PNG using `sharp`. Use escaped text, fixed colors, fixed font stack, bounded column widths, and deterministic dimensions derived from table content. Filename should be stable from a semantic hash:

```text
portable-table-<sha256(canonical-table-json).slice(0, 12)>.png
```

In Medium/Substack paste output, replace the source Markdown table with:

```markdown
![<alt text>](https://<site>/writing/<slug>/assets/portable-table-<hash>.png)
```

Do not change the canonical MDX body. The table image is a deterministic distribution artifact derived from the evaluated draft or, for published-post backfill, from the hub-site MDX. It is not a reviewed draft input and must not mutate `.blog-agent/drafts/<slug>/index.mdx` or `.blog-agent/drafts/<slug>/assets/` after evaluation.

Table parser scope:

- Support leading/trailing optional pipes.
- Support alignment separators (`---`, `:---`, `---:`, `:---:`).
- Support escaped pipes (`\|`) and inline-code spans containing pipes.
- Reject malformed separators and ragged rows unless missing trailing cells can be unambiguously padded.
- Do not convert tables inside fenced code blocks.
- Do not convert pipe-looking rows inside blockquotes or list items in the first implementation.
- Include enough alt text for external platforms: `Table: <nearest preceding heading or post title>`.

## Adversarial Review

**Tier**: 2
**Reviewers**: Architect sub-agent + code-reviewer sub-agent
**Refinement cycles**: 1
**Attack framework**: Hidden assumptions, overcomplication, scope drift, weak success criteria, verification-loop readiness, reasoning divergence

### Critical

1. Backfill source was underspecified. Published-post backfill could incorrectly read stale or missing `.blog-agent/drafts/<slug>/index.mdx` because current Medium/Substack generators read draft-only paths. Resolution: plan now requires sourceMode `backfill` to read hub-site `content/posts/<slug>/index.mdx` first, fallback only when appropriate, and test stale draft cases.
2. Manifest path bases conflicted with existing kit-root-relative text paths. Resolution: plan now defines text entries as kit-root-relative filenames and generated image entries as post-root-relative `assets/...` paths with explicit local/site resolvers.
3. Complete-bundle invariant conflicted with `social.distribution_kit.enabled` and `persist_to_site`. Resolution: plan now defines Medium/Substack persistence as controlled by `publish.medium`/`publish.substack`; `social.distribution_kit.enabled=false` suppresses LinkedIn/Hacker News artifacts only; `persist_to_site=false` suppresses post-preview direct push, not the reviewed site PR bundle.
4. Table assets were initially planned under `.blog-agent/drafts/<slug>/assets/` after evaluation. Resolution: table assets now generate under `.blog-agent/social/<slug>/assets/` as deterministic derived artifacts and are copied into site `assets/`; publish must not mutate evaluated draft MDX or draft assets.
5. Paste generators were write-only, risking timestamp churn before manifest reuse checks. Resolution: plan now requires pure render/derive helpers plus a single write phase after manifest invalidation.
6. Post-preview paste steps could regenerate artifacts outside preview review. Resolution: plan now requires paste steps to be load-only when a manifest exists and to fail on missing/mismatched artifacts.
7. Step 11 could overwrite reviewed artifacts after merge. Resolution: plan now requires site manifest/artifact byte comparison before direct-push persistence and fail-closed behavior on conflicts.

### Consider

1. Table rendering is a second subsystem beyond paste persistence. Kept in scope because the user's stated problem includes not being able to create tables for Medium/Substack, but the plan isolates it in `table-assets.ts` with focused tests.
2. Table PNGs reduce editability/accessibility versus raw Markdown. The plan now requires meaningful alt text and keeps canonical MDX unchanged so the hub retains the real table source.
3. Parser scope needed tightening. The plan now enumerates supported and rejected pipe-table cases for first implementation.
4. Manual verification used a potentially stale global `blog` binary. The plan now uses `node dist/cli/index.js`.

### Reasoning Divergence

1. Alternative: split durable paste persistence and table PNG rendering into separate plans. Not chosen because the operator-facing issue is one publication-bundle gap: the post directory should contain everything needed for distribution, including portable table assets.
2. Alternative: introduce a second manifest for Medium/Substack. Not chosen because one manifest is easier to validate and persist safely, as long as path bases are explicit.

## Notes

- This plan intentionally avoids adding new runtime dependencies. `sharp` is already in use and can render deterministic SVG table markup to PNG.
- This plan avoids changing the published canonical MDX body to replace tables. Table PNGs are distribution artifacts derived from the evaluated draft.
- This plan prefers extending the existing distribution kit manifest over adding a second manifest, because publish already has manifest reuse, hash validation, and site persistence around that object.
- The riskiest implementation point is timing: complete bundle files need to exist before `site-pr` / `site-update` copy assets and open the preview PR, while existing pipeline step rows still expect visible `paste-medium` and `paste-substack` steps later in the cycle.
