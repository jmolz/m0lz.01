# Feature: Platform Article Images

> Generated by the migrated `plan-feature` command. Read `AGENTS.md` and referenced files before implementing.

## Overview

Add deterministic platform-image generation to `m0lz.01` so every publishable draft can produce the article assets needed for Dev.to, Medium, and Substack distribution:

- Dev.to cover image: `1000x420` PNG, 50:21.
- Medium featured image: `1200x675` PNG, 16:9.
- Substack article preview image: `1200x630` PNG, 1.91:1.

The already-published `m0lz-01-launch` article can remain as-is until it enters an explicit update cycle. The original article has legacy image fields in both the local draft workspace and the `m0lz.00` site content directory:

- `/Users/jacobmolz/blog/.blog-agent/drafts/m0lz-01-launch/assets/devto-cover.webp`
- `/Users/jacobmolz/code/m0lz.00/content/posts/m0lz-01-launch/assets/devto-cover.webp`

The feature belongs in `m0lz.01`, not just as a one-off image edit. The publish pipeline must generate or validate these assets before `site-pr` / `site-update` copies draft assets into `m0lz.00`, otherwise later cross-post files can point at URLs that were never shipped.

Medium's Help Center confirms a featured image must already be added to the story before it can be selected as featured, and Medium image layout options require at least `1192px` width. The `1200x675` target satisfies that width and gives a stable 16:9 asset. Substack's Help Center distinguishes the `1100x220` email banner from post/social previews; the per-article Substack generated asset is the `1200x630` preview image.

## User Story

As Jacob, I want `m0lz.01` to generate Dev.to, Medium, and Substack-ready article image assets from one visual framework, so that the canonical site, paste-ready cross-post files, and platform upload steps stay consistent without manual image resizing after publish.

- [ ] `m0lz.01` can generate Dev.to (`1000x420` PNG), Medium (`1200x675` PNG), and Substack preview (`1200x630` PNG) assets for a draft slug.

- [ ] Existing published articles, including `m0lz-01-launch`, are left untouched unless they enter an explicit update cycle.

## Adversarial Review

**Tier**: 2
**Reviewers**: Local planner self-audit only. The repo command requests independent adversarial streams, but current tool policy permits spawned agents only when the user explicitly asks for subagents. Do not treat this as a substitute for post-implementation `.codex/commands/evaluate.md`.
**Refinement cycles**: 1

### Critical

1. Generation after paste would create broken public URLs - resolution: Task 6 makes `ensurePlatformImages` run inside `createSitePR` before assets are copied, which covers both initial and update mode because `site-update` delegates to `createSitePR`.
2. A one-off image edit would not satisfy "part of m0lz.01" - resolution: Tasks 3-8 add reusable core, CLI, pipeline, paste, and frontmatter support; Task 9 handles only the latest-article backfill.
3. Remote AI image generation in publish would make the pipeline non-deterministic - resolution: Task 2 and Task 3 use local `sharp` transforms and a deterministic fallback template, with no remote image API dependency.
4. Adding new frontmatter fields can be silently stripped by regeneration - resolution: Task 4 covers parse/serialize and `regenerateDraftFrontmatter` preservation.

### Consider

1. `sharp` is a native dependency and increases install surface. It is justified because the CLI must decode WebP and emit exact PNG sizes cross-platform; shelling out to macOS-only `sips` or relying on ImageMagick would be less portable.
2. Substack's `1100x220 PNG` recommendation is for email/header banners, while Substack also documents different social/post preview dimensions. The plan intentionally targets the user's requested header/banner asset and does not add a separate Substack social preview image.
3. The deterministic fallback image may be less visually polished than a hand-authored cover. The plan keeps explicit source-image fields and existing `devto_main_image` as the preferred source so authored artwork wins when present.

### Reasoning Divergence

1. Alternative: Add only frontmatter fields and let the operator create images manually. Rejected because it fails the "easy and straightforward" requirement and does not prevent stale or missing hosted assets.
2. Alternative: Generate images during Medium/Substack paste steps. Rejected because those steps run after `site-pr`, creating a real risk of paste files referencing assets that were never copied to `m0lz.00`.
3. Alternative: Add platform-specific API upload automation. Rejected because Medium/Substack workflows here are intentionally paste/manual and the current pipeline avoids blocking on unreliable or unavailable publishing APIs.
