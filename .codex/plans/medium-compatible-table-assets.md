# Feature: Medium-Compatible Table Assets

> Generated by the migrated `plan-feature` command. Read `AGENTS.md` and referenced files before implementing.

## Overview

Fix cross-post table handling so publication bundles produce readable, platform-compatible table assets and operator handoff files that work with Medium's and Substack's real editor constraints.

The current implementation solves the right class of problem but with the wrong publication contract:

- `src/core/publish/table-assets.ts` converts Markdown pipe tables into PNGs, but the renderer uses fixed 42px rows and one SVG `<text>` node per cell. Long cells overflow across columns, which matches the screenshot failure.
- `medium-paste.md` and `substack-paste.md` currently reference table images through public `https://m0lz.dev/writing/<slug>/assets/portable-table-<hash>.png` Markdown image links. The live URL is valid and served as `image/png`, but Medium's editor does not treat arbitrary image URLs as uploadable image blocks.
- Medium embeds are provider-based through Embed.ly, not arbitrary file embeds. Medium's documented image workflow is upload or drag/drop from desktop. Its documented import workflow can import a canonical URL and automatically apply the canonical link, but that still needs operator verification for tables.
- Substack likewise does not support Markdown in its post editor, does not support custom CSS/HTML, and documents images as upload or drag/drop. Its rich-media URL embeds are provider-style embeds, not a durable table-image contract.

The durable strategy is:

1. Keep canonical MDX semantic and unchanged.
2. Generate high-quality deterministic table images for platform handoff.
3. Stop treating Medium table images as remote embeds.
4. Persist a clear upload/import checklist with exact local asset paths, alt text, captions, and canonical URL instructions.
5. Verify the generated images structurally and visually enough that unreadable table PNGs cannot pass the plan contract.

## User Story

As a technical author publishing through `blog`, I want Medium and Substack distribution artifacts to include readable table images and clear upload instructions, so that cross-posting does not require hand-fixing broken Markdown tables, rejected embeds, or unreadable screenshots.

## Success Criteria

- [ ] Long evidence/boundary-style table cells wrap inside their columns instead of overlapping adjacent cells.
- [ ] Generated table PNGs are sized for Medium/Substack upload, with width, row height, padding, and font sizes suitable for article publication.
- [ ] Medium handoff no longer requires arbitrary `m0lz.dev` table image URLs to be accepted as embeds.
- [ ] Medium handoff recommends the official import-from-canonical-URL path first, then local image upload fallback for table assets.
- [ ] Substack handoff treats table assets as local image uploads, not Markdown table syntax, custom HTML tables, or public `portable-table` URL embeds.
- [ ] Manifest provenance still covers every generated table image and every text handoff artifact.
- [ ] Canonical MDX and evaluated draft assets remain unchanged by cross-post table generation.
- [ ] Existing table detection behavior remains intact for fenced code, blockquotes, list-like text, malformed tables, deterministic filenames, and stale generated cleanup.
- [ ] All validation commands pass.

### Current Failure Points

- `textWidth(value)` caps every cell at 280px and estimates width with `plain.length * 7 + 32`; it does not wrap.
- `renderSvg(table)` fixes `rowHeight = 42` for every row, regardless of content length.
- Each cell writes one `<text>` node at `y + 26`, so long text has no line breaks, no per-cell clipping, and no dynamic row height.
- `derivePortableTables(...)` always replaces the table with `![alt](https://.../assets/portable-table-<hash>.png)`, which is not the same thing as a Medium image upload.
- Current tests prove the PNG is valid and deterministic, but they do not prove text fit, wrapping, Medium handoff usability, or upload instructions.

## Strategic Design

### Core Decision

Keep image-backed table assets, but treat them as platform-upload artifacts, not arbitrary embeds.

Rationale:

- Medium does not expose a reliable native table block in the documented story editor.
- Medium's embed flow is provider-gated and does not make a plain PNG URL into an inline image block.
- Medium's image workflow is local upload/drag/drop; the pipeline should produce local upload assets and precise instructions.
- Importing the canonical hub URL is still the best first Medium workflow because it automatically applies canonical metadata, but imported tables/images must be checked and corrected with local table assets if Medium loses fidelity.
- Substack is stricter than Medium for this feature: no Markdown, no custom HTML/CSS, and documented image insertion through upload or drag/drop. Public table PNG URLs may remain useful for reference, but successful Substack publication must not depend on pasting those URLs as embeds.
- Generated images are acceptable for cross-post surfaces only if the canonical hub article remains semantic and the handoff includes alt/caption guidance.

### Target Artifact Layout

Keep existing bundle paths and add platform handoff guidance:

```text
content/posts/<slug>/
  index.mdx
  assets/
    medium-featured.png
    substack-preview.png
    portable-table-<hash>.png
  distribution/
    medium-paste.md
    medium-upload-checklist.md
    substack-paste.md
    substack-upload-checklist.md
    manifest.json
```

The exact checklist filenames may be adjusted during implementation if the manifest model strongly prefers a single `crosspost-upload-checklist.md`, but the contract requires platform-specific instructions for Medium and Substack.

### Table Image Layout Contract

Use deterministic SVG rendered through existing `sharp`.

Target image profile:

- Minimum display width: `1192px` for Medium placement compatibility.
- Raster density: render at 2x or an equivalent high-resolution profile while keeping file size below Medium's documented 25MB image limit.
- Background: white.
- Header: strong contrast, readable at Medium column sizes.
- Body: alternating row fills, clear borders, no decorative gradients.
- Font size: publication-readable, not compact dashboard text.
- Padding: enough horizontal and vertical padding to avoid cramped cells.

Layout algorithm:

- Parse Markdown tables with the existing detection rules.
- Normalize display text by removing lightweight Markdown table syntax noise while preserving visible cell content.
- Compute column widths against a fixed table canvas instead of letting long cells expand the canvas.
- Allocate wider columns to text-heavy headers such as `Evidence`, `Boundary`, `Result`, `Notes`, and `Rationale`, while retaining minimum widths for short identifier columns.
- Wrap text by words, with hard breaks for single long tokens that exceed the cell width.
- Render wrapped lines as `<tspan>` elements.
- Compute each row height from the maximum wrapped-line count across that row.
- Reject or visibly mark impossible tables only if the table exceeds an explicit maximum column count; do not silently produce overlapping text.

Accessibility and provenance:

- Keep canonical MDX tables semantic on m0lz.dev.
- Generate concise image alt text and captions for Medium/Substack upload instructions.
- The checklist must include the canonical URL as the accessible/full-fidelity table source.
- Manifest entries continue recording table source hash, row count, column count, PNG hash, dimensions, and byte size.

## Adversarial Review

**Tier**: 2
**Reviewers**: isolated code-reviewer subagent plus local planning pass
**Refinement cycles**: 2 planning refinements, then repeat execute/evaluate until contract pass

### Critical

1. Native Medium and Substack tables are not reliable targets. Resolution: the plan treats cross-post tables as generated image uploads with canonical semantic source retained on m0lz.dev.
2. Public image URLs are valid but the editor workflow is wrong. Resolution: the plan stops making Medium table success depend on arbitrary URL embed acceptance.
3. Substack has the same failure mode as Medium, plus no Markdown/custom HTML escape hatch. Resolution: the plan explicitly requires local upload guidance and forbids Substack success from depending on raw Markdown, HTML tables, or public table-image URL embeds.
4. Image tables are an accessibility regression if used as the only source. Resolution: canonical MDX remains semantic, and upload checklists require alt/caption text plus canonical source reference.
5. New checklist files can easily bypass this repo's strict manifest/copy allowlists if treated as sidecars. Resolution: the plan names the manifest schema, `input_hash`, validation, text enumeration, fixed filename allowlist, expected site path, and stale cleanup surfaces that must be updated.
6. Visual quality cannot be proven by `metadata.width > 0` or `<tspan>` existence. Resolution: add layout-bound metrics or parseable SVG-bound tests for long-cell, long-token, many-column, and maximum-column fixtures, plus manual latest-content PNG inspection.

### Consider

1. A future browser-based rich-copy preview could reduce manual upload work, but it would be a separate feature because Medium editor behavior is authenticated and not deterministic enough for this publish gate.
2. A future Datawrapper/Airtable path could preserve interactive tables, but it introduces external accounts/providers and does not fit the current deterministic offline publish rule.
3. A future Substack-specific rich-text export could improve paste ergonomics, but it must be proven against the authenticated editor and email rendering before replacing local image upload guidance.
4. Latest-content backfill is execution-sensitive. Use the freshly built repo CLI or prove the `blog` binary points at this checkout before regenerating bundles, so stale global installs cannot produce old handoff behavior.

### Reasoning Divergence

1. Alternative: keep remote image URLs and tell the user to paste Markdown image syntax. Rejected because Medium's and Substack's documented image workflows are upload/drag/drop, Substack does not support Markdown in the post editor, and the user has already observed URL rejection on Medium.
2. Alternative: convert tables to raw HTML. Rejected because Medium/Substack editor docs do not provide a durable custom HTML table contract, and Substack explicitly rejects custom HTML/CSS in the editor.
3. Alternative: screenshot the hub page table. Rejected because deterministic SVG/Sharp rendering is easier to test, hash, regenerate, and keep independent of browser layout.

## Notes

- The live `m0lz.dev` asset URL checked during planning returned `HTTP/2 200`, `content-type: image/png`, `content-disposition: inline`, and CORS `*`. That confirms the route is not the primary blocker.
- The user-facing problem has two independent causes: poor image generation and incorrect Medium handoff assumptions. Both must be fixed for the contract to pass.
- This plan intentionally avoids new runtime dependencies and remote platform APIs.
