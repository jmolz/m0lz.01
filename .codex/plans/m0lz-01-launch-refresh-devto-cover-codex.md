3. Hidden assumption: the research branch icon is a page-component bug. Resolution: Task 7 updates m0lz.00 `data/projects.ts`, and Tasks 4-5 fix future m0lz.01 generated research frontmatter.
4. Overcomplication risk: adding OpenAI image generation to `blog publish` would add a new API-key dependency and nondeterminism. Resolution: this plan keeps image generation as an authoring artifact and only adds deterministic `main_image` consumption to publish.

### Consider

1. The Codex-native `/blog` surface can be either a real `.agents/skills/blog` implementation now or a documentation/content positioning pass that explains CLI use from Codex. If the implementation chooses docs only, the article must not imply a packaged Codex plugin exists.
2. A future feature can automate cover generation as a separate `blog visuals cover` command with explicit API-key setup, image optimization, and file-size validation.

### Reasoning Divergence

1. Alternative: reset the publish pipeline and rerun site-pr from scratch. Why not: the site PR was already merged, the canonical URL is live, and rerunning the full initial path risks duplicate PR/state churn. A content refresh plus local draft sync preserves the existing resume path and directly fixes the missing Dev.to step.
2. Alternative: host Dev.to covers under m0lz.00 `public/social/`. Why not: post-local assets already copy through the m0lz.01 site-pr path and produce stable public URLs under the canonical article, keeping the asset attached to the content it represents.
