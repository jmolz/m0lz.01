# Feature: Distribution Copy and LinkedIn Image Quality

> Generated by the migrated `plan-feature` command. Read `AGENTS.md` and referenced files before implementing.

## Overview

Fix the durable distribution-kit outputs so they are ready to post, not just mechanically complete.

The current Stack Loops bundle proves three defects in the generator contract:

- `linkedin.md` leaks operator-only lines: `Image prompt: ./distribution/linkedin-image-prompt.md` and `Alt text: ...`.
- LinkedIn, Hacker News, and Substack copy can hard-cut frontmatter descriptions with `...`, producing fragments such as `and...`.
- The default LinkedIn image path is `prompt-only`, so a normal bundle can ship no `assets/linkedin-feed.png`; the prompt that is written describes a generic desk scene that is not strong enough for the product being launched.

This feature makes public distribution artifacts audience-facing, gives platform descriptions natural stopping points, and adds a deterministic LinkedIn feed card image mode so latest-content bundles produce a real visual without depending on OpenAI image generation. The default image change is in scope because the user's current failure is not just weak prompt copy; the generated bundle in the hub has no LinkedIn feed image at all.

## User Story

As the author publishing a technical launch post, I want the generated LinkedIn, Hacker News, Substack, and bundle image assets to be coherent and post-ready, so that the final repo contains polished distribution assets instead of internal prompts, clipped descriptions, or generic desk visuals.

## Success Criteria

- [ ] `linkedin.md` contains public-facing post copy only: no `Image prompt:`, no `Alt text:`, no relative asset paths, no "durable distribution kit" fallback line.
- [ ] LinkedIn copy has a pinned structure, a local total-post limit, and never hard-clips the post description with `...`.
- [ ] Hacker News first-comment copy uses the same no-hard-ellipsis copy fitter.
- [ ] Substack subtitle stays within the configured subtitle limit and never ends with `...`; if no complete sentence fits, generation fails before site checkout/copy/commit with an actionable error instead of writing bad copy.
- [ ] Default distribution-kit generation produces a real `assets/linkedin-feed.png` using a deterministic local card renderer and records it in `manifest.json`.
- [ ] `local-card` writes no `linkedin-image-prompt.md`; stale site prompt artifacts are cleaned up when moving from prompt-based modes to local-card.
- [ ] `prompt-only` remains available as an explicit compatibility mode and still writes a prompt without calling an image provider or writing an image.
- [ ] `generate` and `required` remain available for OpenAI image generation, but their prompt no longer asks for a desk, notes, or terminal-window scene.
- [ ] The Stack Loops bundle is regenerated from the fixed generator and synced into the hub repo so main contains the final LinkedIn text, Substack paste, manifest, and LinkedIn image asset.
- [ ] All validation commands pass.
- [ ] No regressions in manifest hash verification, site persistence, stale cleanup, or existing Medium/Substack table upload guidance.

### Documentation

- https://www.linkedin.com/help/linkedin/answer/a528176 - LinkedIn Help currently documents a 3000-character post limit, so the generator should not use a 220-character hard clip for the public LinkedIn body.
- https://support.substack.com/hc/en-us/articles/360039016992-How-do-I-edit-what-my-post-looks-like-on-social-media - Substack support documents editable social preview text and recommends preview images at least 1200 x 630. The repo's 120-character Substack subtitle cap is a local product constraint, not an official hard limit.
