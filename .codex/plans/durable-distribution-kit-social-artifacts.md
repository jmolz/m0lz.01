# Feature: Durable Distribution Kit and LinkedIn Visuals

> Generated by the migrated `plan-feature` command and revised after adversarial plan review.
> Status: finalized implementation plan. Contract approved by user instruction: "Revise the plan and finalize it."

## Overview

The publish pipeline already generates LinkedIn and Hacker News paste files in `.blog-agent/social/<slug>/`, but those files are runtime artifacts only. They are not committed to the canonical hub repo, they are not reviewable later, and the current copy quality can be poor. For `m0lz-01-launch`, the current generated outputs prove the gap:

- `linkedin.md` says `Key takeaway: m0lz.` because sentence extraction splits on `m0lz.01`.
- `hackernews.md` uses `Show HN: m0lz-01-launch` instead of the real post/product title.
- `hackernews.md` duplicates `Companion repo: n/a`.
- Neither file is present in `/Users/jacobmolz/code/m0lz.00/content/posts/m0lz-01-launch/`.

This feature turns social output into a durable per-post distribution kit. The kit includes polished LinkedIn copy, Show HN submission copy, a LinkedIn image prompt, an optional GPT Image 2 generated LinkedIn feed image, and a manifest with artifact hashes/provenance.

The final design resolves the adversarial review findings by splitting the work into two phases:

1. **Local generation/reuse before site mutation**: before `site-pr` or `site-update` copies anything into m0lz.00, generate or reuse the local distribution kit. Required image mode fails here, before any site repo mutation.
2. **Post-preview persistence only**: the existing `social-text` pipeline step becomes a persist-only step that copies already-generated artifacts into the hub repo after the preview gate and frontmatter/README updates. It must never call OpenAI.

The generated LinkedIn image is a fixed PNG artifact at `assets/linkedin-feed.png`. The config does not expose arbitrary output formats in v1, because a configurable format conflicts with the fixed public filename and site content-type contract.

## User Story

As a technical content author, I want each published post to leave behind a durable distribution kit, so that LinkedIn and Show HN launch materials are as inspectable and reproducible as the article, images, research page, and cross-post paste files.

## Success Criteria

- [ ] `blog publish start <slug>` and `blog update publish <slug>` generate or reuse a local distribution kit before `site-pr` / `site-update` mutates the site repo.
- [ ] Required LinkedIn image generation fails before any site repo checkout, copy, commit, push, or PR when `OPENAI_API_KEY` is missing.
- [ ] Re-running local kit generation with unchanged metadata/prompt/model/size reuses existing artifacts, preserves manifest timestamps, does not call the image provider, and produces no new site diff.
- [ ] The existing `social-text` step persists already-generated artifacts to `content/posts/<slug>/distribution/` and never calls OpenAI.
- [ ] LinkedIn output contains title, description, canonical URL, tags, image path, and alt text; it contains no placeholders, no `n/a`, no emojis, and no broken fragment such as `m0lz.`.
- [ ] Hacker News output uses `Show HN:` only for `project-launch`, derives the title from frontmatter when available, keeps the title <=80 characters, and contains at most one companion repo line.
- [ ] Backfill for already-published posts reads hub repo frontmatter first, draft frontmatter second, and DB row last.
- [ ] `m0lz.00/content/posts/m0lz-01-launch/` can be backfilled with `distribution/` files and `assets/linkedin-feed.png`.
- [ ] All validation commands pass.
- [ ] Docs and regression-suite inventories mention the new tests and CLI surface.

## Design Constraints

- Internal TypeScript imports must use `.js` extensions.
- No OpenAI calls in tests. Tests inject a fake provider.
- Default package behavior must not require network or `OPENAI_API_KEY`.
- The generated LinkedIn image format is fixed to PNG in v1.
- Generation is content-addressed by metadata hash + prompt hash + image config hash.
- `force` is the only path that regenerates an existing matching artifact.
- Post-preview persistence copies already-generated files only. It cannot generate or call OpenAI.
- Direct pushes to `main` must use strict direct-push guardrails, not a loose `git add .`.
- Backfill source precedence differs from normal publish:
  - normal publish/update: evaluated draft frontmatter first, DB row fallback
  - backfill: hub repo frontmatter first, draft frontmatter second, DB row fallback
- Update-aware copy must use the open update cycle's `cycle_number`, `summary`, and `opened_at` when available. It must not invent an update narrative when the summary is absent.

## Testing Strategy

### Unit Tests

- Where: `tests/publish-social.test.ts`, `tests/distribution-kit.test.ts`
- Pattern: existing temp dirs, in-memory DB, real templates for normal paths, local fixture templates for error paths, fake image provider injection.
- Minimum: metadata precedence by mode, project-launch HN title, no duplicate companion repo, no broken decimal/product ID takeaway, no emoji, prompt required/banned terms, hash-stable manifest reuse.

### Integration Tests

- Where: `tests/publish-distribution-kit.test.ts`, `tests/publish-pipeline.test.ts`, `tests/update-publish-pipeline.test.ts`
- Pattern: temp site repo, in-memory or temp SQLite, fake image provider injection, subprocess spies/mocks, path-scoped staging assertions.
- Minimum:
  - prompt-only mode writes prompt + manifest but no image.
  - fake generated mode writes PNG and hashes it.
  - required mode fails without `OPENAI_API_KEY` before any site repo mutation.
  - second run with unchanged inputs is a no-op and does not invoke provider.
  - `social-text` persists only and never calls provider.
  - site persistence refuses unrelated dirty/staged files.
  - strict ahead-commit replay accepts only exact distribution commit shape.
  - no network calls occur in tests.

## Adversarial Review

**Tier**: 2
**Reviewers**: Codex GPT-5.5 xhigh external task + local Codex plan reviewer. Claude opus Task stream was unavailable in this Codex tool environment.
**Refinement cycles**: 1
**Attack framework**: Karpathy four principles + PoetiQ cross-model verification

### Critical (resolved before finalizing)

1. Verification-loop readiness: `manifest.json` timestamp + nondeterministic GPT image generation could create a new image/hash/commit on every resume.
   Resolution: Task 5 requires content-addressed input hashes, artifact hash verification, provider reuse, preserved timestamps on reuse, and tests proving second-run no-op behavior.

2. Hidden assumption: required image mode would fail only after prior site/cross-post side effects if left in Step 11.
   Resolution: Tasks 5 and 6 move generation/reuse before `site-pr` / `site-update` mutation. Task 8 makes `social-text` persist-only and forbids OpenAI calls.

3. Hidden assumption: backfill could trust draft frontmatter for already-published posts.
   Resolution: Tasks 2 and 9 define backfill metadata precedence as hub repo frontmatter first, draft second, DB last.

4. Hidden assumption: direct-push trust boundaries were under-specified.
   Resolution: Task 7 adds `site-artifacts.ts` with strict direct-push guardrails modeled on `frontmatter.ts` / `readme.ts`, including clean-index, dirty-file, origin, path-scoped staging, no-op, and crash-replay tests.

5. Weak success criteria: "human-quality launch framing" and "LinkedIn psychology" were subjective.
   Resolution: Success criteria, Tasks 3 and 5, and the contract now use concrete assertions: no placeholders, no `n/a`, no emojis, no duplicate repo line, HN <=80 chars, exact image dimensions, hashes, and prompt required/banned terms.

6. Weak success criteria: docs task omitted `.windsurf/workflows/review.md`.
   Resolution: Task 11 and contract criterion 10 include `.windsurf/workflows/review.md`.

### Consider (addressed or intentionally scoped)

1. Overcomplication: provider/output format config before multiple providers existed.
   Decision: v1 fixes provider to OpenAI and output format to PNG. Only mode/model/size/quality are configurable.

2. `output_format` conflicted with fixed `linkedin-feed.png`.
   Decision: remove `output_format` from the v1 config and contract.

3. "Update-aware copy" was unspecified.
   Decision: Task 8 requires open update cycle `summary`, `cycle_number`, and `opened_at`; absent summary emits a fixed `(no update summary provided)` string.

4. OpenAI access can fail despite `OPENAI_API_KEY`.
   Decision: Task 10 allows a prompt-only launch-post fallback and requires manifest mode/provenance to record the path used.

5. Backfill mutates m0lz.00.
   Decision: production backfill is explicitly in scope and authorized by this finalized plan after validation passes.

### Reasoning Divergence

1. Extend `platform-images.ts` instead of creating distribution-specific image code.
   Decision: reuse its patterns and pre-site mutation timing, but keep LinkedIn distribution prompt/image provenance in `distribution-kit.ts` because it is social/distribution-specific and content-addressed differently.

2. Split local generation and persistence.
   Decision: adopted.

3. Ship prompt-only v1 and defer image generation.
   Decision: not adopted globally because the user explicitly asked for GPT Image 2, but prompt-only remains the safe default and the launch backfill has a documented fallback.
