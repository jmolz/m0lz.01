# Feature: Benchmark Results Phase Gates

> Generated by the migrated `plan-feature` command. Read `AGENTS.md` and referenced files before implementing.

## Overview

The live dogfood flow exposed benchmark/evaluate phase contract failures:

1. `blog benchmark run --results-file .../environment.json` accepted an environment snapshot as benchmark results.
2. `blog benchmark complete` advanced a post to `draft` even though no valid benchmark evidence existed.
3. `blog evaluate structural-autocheck` could write a clean-looking lint artifact before the active evaluation workflow existed.

The fix must cover prevention and recovery. A released CLI already let the real `/Users/jacobmolz/blog` workspace reach `draft` with invalid benchmark state, so operators need a supported repair path that preserves raw artifacts and avoids direct SQLite edits.

The deeper invariant is provenance: DB benchmark row, canonical `results.json`, environment snapshot, draft content, and structural-autocheck sidecar must describe the same lifecycle event or active evaluation cycle.

## User Story

As a blog operator dogfooding `blog` on a project-launch post, I want bad benchmark artifacts rejected and already-corrupted state repairable through explicit CLI commands, so that I can continue the post lifecycle without hand-editing `.blog-agent/state.db` or trusting stale "all clear" output.

## Success Criteria

- [ ] `blog benchmark run <slug> --results-file <file>` validates benchmark-result input and rejects `environment.json`.
- [ ] Rejected benchmark imports mark the run row `failed` and do not overwrite canonical `.blog-agent/benchmarks/<slug>/results.json`.
- [ ] Results whose embedded `slug` does not match the command slug are rejected.
- [ ] User-supplied results input does not require `run_id`; the CLI writes the authoritative DB run id into the canonical stored result.
- [ ] `blog benchmark complete <slug>` cannot advance without a valid canonical `results.json` and a completed benchmark row whose id matches the canonical `run_id`.
- [ ] `blog benchmark show <slug>` reports invalid canonical results as `invalid (...)` instead of presenting them as usable evidence.
- [ ] Optional project-launch benchmark attempts can be skipped after a bad run from either `benchmark` or `draft`, with raw artifacts preserved and an audit receipt written.
- [ ] Optional-skip repair refuses to proceed when an existing draft file may contain stale benchmark-derived prose.
- [ ] Draft commands do not silently consume malformed benchmark results; they show invalid state or fail with the next repair command.
- [ ] Structural autocheck is phase-gated and cycle-fresh: stale sidecars from older/bad runs are not accepted for the active evaluation cycle.
- [ ] Every benchmark result consumer is inventoried and either routed through the validator or intentionally documented as raw-file-only.
- [ ] README, `AGENTS.md`, and `/blog` skill docs explain the real flow, the `environment.json` mistake, and the repair path.
- [ ] Focused regression tests and the full validation chain pass.

2. Optional skip could leave stale benchmark prose in an existing draft. Section: Task 6. Resolution: skip repair refuses existing draft files until a separate draft-body repair exists.

4. `run_id` semantics were impossible for operators. Section: Tasks 2-3. Resolution: user input no longer requires `run_id`; canonical storage writes the DB run id.
5. Completion validity was split between CLI and core. Section: Task 4. Resolution: `completeBenchmark` now accepts `benchmarkDir` and validates canonical evidence in core.

### Consider (acted on or explicitly scoped)

1. `blog update benchmark` is another ingestion surface. Resolution: Task 7 validates update-cycle results through the same helper.
2. Research-page summary parsing could go stale under the new canonical shape. Resolution: Task 9 updates the consumer and tests canonical summary behavior.

3. Alternative: treat structural autocheck as phase-only. Rejected because the dangerous artifact is stale-cycle trust, not only early command execution.
