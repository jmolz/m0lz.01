# Feature: Phase 7 — Post Lifecycle (Unpublish, Update, Orchestration)

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Phase 7 closes the content lifecycle loop. Phase 1–6 take a post from idea to
published; Phase 7 handles everything after: rolling a published post back,
re-running benchmarks against current versions and republishing with an update
notice, and orchestrator skills that drive the full pipeline end-to-end.

Architecturally, Phase 7 adopts a **hybrid pipeline model** converged on by two
independent adversarial reviewers (Claude architect + GPT-5.4 high):

1. **Updates are a mode of the publish pipeline**, not a third pipeline.
   `PipelineContext.publishMode: 'initial' | 'update'` dispatches step behavior.
   The publish pipeline-runner, lock semantics, stale-running reclaim, and
   config reconciliation from Phase 6 apply to updates automatically — one
   codepath, not two. An `update_cycles` table tracks open cycles as
   first-class rows; `posts.phase` stays `published` throughout an update
   (no backward transitions, no `update_in_progress` flag, no phase-boundary
   overrides in benchmark/draft/evaluate).

2. **Unpublish is its own pipeline** because it is inverse-shaped (removal,
   not creation) and one-shot per post. Separate `unpublish_steps` table,
   prefixed step names (no collision with publish), PR-only site revert,
   reuses the same per-slug lock as publish to guarantee mutual exclusion.

3. **Three orchestrator skills** (`/blog-pipeline`, `/blog-update`,
   `/blog-unpublish`) document the narrative flow in Claude Code sessions.

Phase 7 preserves every Phase 6 invariant: lock-held-across-completion,
probe-then-mutate, per-step URL persistence via COALESCE, index-cleanness
guards before direct-push, origin-URL trust boundaries, stale-running
reclaim, and config-driven step reconciliation on resume.

## User Stories

**Unpublish:**

> As a content author, I want `blog unpublish start <slug> --confirm` to
> remove a published post from my site, unpublish the Dev.to cross-post, and
> generate manual-removal instructions for Medium and Substack, so that I can
> retract content cleanly when I change my mind without leaving orphaned
> copies across platforms.

**Update:**

> As a content author, I want `blog update start <slug> --summary "..."` to
> open an update cycle and then `blog update benchmark/draft/evaluate/publish`
> to re-run my benchmarks against current software versions, regenerate the
> draft with an update notice, re-evaluate through a fresh review cycle, and
> republish (MDX body + Dev.to PUT), so that my benchmark posts stay accurate
> over time without losing the original data readers cited and without the
> canonical site ever going out of sync with Dev.to.

**Orchestration:**

> As a content author, I want `/blog-pipeline` to walk me from a topic
> through research, benchmarking, drafting, evaluation, and publish in one
> skill session, and `/blog-update` and `/blog-unpublish` to do the same for
> updates and rollbacks, so that I operate the pipeline as three intent-clear
> skills instead of a dozen CLI commands.

## Success Criteria

### Unpublish

- [ ] `blog unpublish start <slug> --confirm` on a `published` post executes
      all unpublish steps end-to-end and advances phase to `unpublished`
- [ ] `blog unpublish start <slug>` without `--confirm` exits code 1 before
      any DB mutation or external API call
- [ ] `blog unpublish start <slug>` on a post NOT in `published` phase exits
      code 1 with a descriptive phase-boundary error
- [ ] Re-running `blog unpublish start <slug>` on an already-`unpublished`
      post returns success without mutation or network calls (idempotent)
- [ ] Dev.to unpublish step probes by canonical URL and PUTs
      `published: false` on match; probe-miss is a successful skip
      (not an error)
- [ ] Site-revert step creates a PR against the site repo that sets
      `published: false` in the post's frontmatter (PR-only, no direct-push
      mode)
- [ ] Project README link removal is idempotent: skips with a descriptive
      reason when the link is already absent or the post has no
      `project_id`
- [ ] Medium and Substack steps generate manual-removal instructions at
      `.blog-agent/social/{slug}/medium-removal.md` and `substack-removal.md`
- [ ] `blog unpublish show <slug>` displays a per-step status table
- [ ] `posts.unpublished_at` is set when the phase advances
- [ ] A metrics row with `event='unpublished'` is written on cycle completion
- [ ] All Phase 6 crash-safety invariants apply: stale `running` rows
      reclaimed at pipeline start, per-step state persistence, lock held
      until completion, direct-push steps assert index clean,
      origin URL verified on every repo touch

### Update

- [ ] `blog update start <slug> --summary "text"` on a `published` post
      opens an `update_cycles` row; the post's `phase` stays `published`
- [ ] `blog update start <slug>` on a post NOT in `published` phase exits
      code 1 with a descriptive phase-boundary error
- [ ] `blog update start <slug>` on a post with an already-open update
      cycle exits code 1 and points to `blog update abort` or
      `blog update publish`
- [ ] `blog update abort <slug>` closes the open update cycle with
      `ended_reason='aborted'`, preserves regenerated artifacts on disk
      for manual inspection, and writes a metrics row
      `event='update_aborted'`
- [ ] `blog update benchmark <slug> --results <file>` records a benchmark
      row with `is_update=1` and `previous_run_id` pointing to the most
      recent prior run for that post; requires an open update cycle
- [ ] `blog update draft <slug>` regenerates the draft MDX in place,
      preserving original body paragraphs and inserting the update notice
      tied to the current cycle_id
- [ ] `appendUpdateNotice` is cycle-keyed (not date-keyed): re-running
      within the same open cycle replaces the notice block for that cycle;
      completed cycles preserve their historical notice blocks
- [ ] `blog update evaluate <slug>` opens a fresh evaluation cycle with
      all new reviewer rows tagged `is_update_review=1` (set explicitly
      from the open update cycle, not inferred from cycle count)
- [ ] `blog update publish <slug>` runs the publish pipeline in
      `publishMode='update'`:
  - `verify` gates on evaluation-passed for the current update cycle
  - `research-page` regenerates (byte-identical output is a no-op commit)
  - `site-update` (update-only step) commits regenerated MDX + assets
    directly to site main via PR (idempotent on re-push)
  - `preview-gate` still pauses for PR merge (update mode also uses PR
    for MDX body changes, mirroring initial publish)
  - `crosspost-devto` probes by canonical → PUTs new body + metadata
    if found; falls through to POST when probe misses (operator
    manually deleted); succeeds either way
  - `paste-medium` / `paste-substack` refresh paste files with update
    framing
  - `companion-repo` is skipped in update mode (already exists)
  - `update-frontmatter` direct-push sets `updated_at`, increments
    `update_count`, refreshes platform URLs
  - `update-readme` is skipped in update mode (link unchanged)
  - `social-text` generates with update framing; summary text pulled
    from `update_cycles.summary` (single source of truth)
- [ ] On `blog update publish` completion: `update_cycles.closed_at` set
      with `ended_reason='completed'`, `posts.update_count` incremented,
      `posts.last_updated_at` set, `posts.phase` remains `published`;
      a metrics row `event='update_completed'` is written
- [ ] `blog publish start <slug>` refuses to run on a post with an open
      update cycle; exits code 1 and directs the operator to
      `blog update publish`
- [ ] `blog update show <slug>` displays cycle state (cycle number,
      summary, opened_at, step status if update-publish started)

#### Task A1: Schema v3 migration

**File:** `src/core/db/schema.ts`, `src/core/db/database.ts`,
`src/core/db/types.ts`
**Action:** MODIFY
**Description:** Bump `SCHEMA_VERSION` from 2 to 3. Add `SCHEMA_V3_SQL`
containing (wrapped in a single transaction with `PRAGMA foreign_keys=OFF`
before and `ON` after, following the SQLite canonical table-rebuild
pattern):

1. Rename `pipeline_steps` to `pipeline_steps_old`.
2. Create new `pipeline_steps` identical to old plus `cycle_id INTEGER
   NOT NULL DEFAULT 0` and `UNIQUE(post_slug, cycle_id, step_name)`.
3. `INSERT INTO pipeline_steps SELECT ..., 0 AS cycle_id FROM
   pipeline_steps_old`.
4. `DROP TABLE pipeline_steps_old`.
5. `CREATE TABLE update_cycles (id, post_slug, cycle_number,
   summary, opened_at, closed_at, ended_reason CHECK IN
   ('completed', 'aborted'))` plus `CREATE UNIQUE INDEX
   idx_update_cycles_open ON update_cycles(post_slug) WHERE closed_at
   IS NULL` (enforces at most one open cycle per post at the DB level).
6. `CREATE TABLE unpublish_steps` (same columns as v2 `pipeline_steps`:
   `id`, `post_slug`, `step_number`, `step_name`, `status`,
   `started_at`, `completed_at`, `error_message`, `UNIQUE(post_slug,
   step_name)`).

Update `PipelineStepRow` in `types.ts` to include `cycle_id`. Add
`UpdateCycleRow` and `UnpublishStepRow` types.

**Tier**: 3

### Critical (resolved before proceeding)

1. **HIDDEN ASSUMPTION — `is_update_review=1` inference** (Claude + GPT).
   Plan relied on `manifest.cycles.length > 1` to flag update reviews,
   but that logic fires only after `rejectEvaluation` closes the prior
   cycle. Update flow never calls reject; the flag would never fire, and
   even if it did, reject-retry cycles would be falsely tagged as
   updates.
   Section: *Cluster D*, *Contract criterion #12*.
   **Resolution:** `initEvaluation` accepts an explicit
   `{ isUpdateReview: boolean }` option; `blog update evaluate` passes
   `true`. Manifest records `is_update_cycle: true`; `recordReview`
   reads it and sets `is_update_review=1` directly. No inference.

2. **CORRECTNESS — update flow never syncs MDX body to site** (GPT).
   Plan only did frontmatter direct-push; regenerated prose/benchmark
   tables would never reach m0lz.dev.
   Section: *Cluster C3 new `site-update` step*.
   **Resolution:** New `site-update` step runs in update mode, commits
   regenerated MDX + assets, opens PR for preview-gate review (mirrors
   initial publish's PR gate).

3. **OVERCOMPLICATION — `site_revert_mode: 'pr' \| 'direct'`** (Claude +
   GPT). YAGNI; also directly contradicted the success criterion
   "site-revert creates a PR, NOT direct-push".
   Section: *Cluster A4* (config) and *Cluster E* (unpublish).
   **Resolution:** Dropped. Unpublish is PR-only. No `site_revert_mode`
   option, no `frontmatter-revert.ts` fallback file.

4. **CORRECTNESS — no guard on `blog publish start` during open update

   Section: *Task C1 publish guard*.

### Reasoning divergence (alternative paths not taken)

   Rejected for this plan because unpublish's external operations
   (PUT published:false, PR that sets published:false, README link

   "inverse" of publish steps would fight the data. Accepted the

   unpublish codepath small (~7 steps, separate table, prefixed names).

2. **`update_cycles` table vs. single boolean `update_in_progress`**

   truth for social-text framing.

3. **Prefix step_name instead of `cycle_id` column** (Claude).
   Rejected: updates repeat the same step names across cycles
   (publish cycle 1's `verify`, cycle 2's `verify`), so prefix
   disambiguation would need a numeric suffix anyway — equivalent
   to a cycle_id column.

## Notes

### Key design decisions

- **`posts.phase` never moves backward during an update.** Open
  cycles are tracked in `update_cycles`; phase-boundary checks in
  benchmark/draft/evaluate modules remain as they are in Phases
  1–5. Update-specific CLI subcommands (`blog update benchmark`,
  `blog update draft`, `blog update evaluate`, `blog update
  publish`) route through the cycle state without touching
  `posts.phase`.

- **Single pipeline-runner; two modes.** Updates reuse every Phase 6
  invariant automatically: stale-running reclaim, config reconcile,
  per-step URL persistence, index-cleanness, origin-match,
  ahead-commit crash replay.

- **Unpublish is separate.** Inverse-shaped, one-shot, different
  step bodies. Parallel `unpublish_steps` table avoids step-name
  collisions with publish. Shares the per-slug lock for mutual
  exclusion.

- **Canonical URL is permanent.** Slugs never reused. Unpublish sets
  `published: false` but does NOT delete the post directory on the
  site. Re-publish later uses the same slug.

- **Explicit is better than inferred.** `is_update_review=1` is set
  by a flag passed through the call chain, not by counting cycles.

### Potential follow-up (out of scope for Phase 7)

- Store `devto_article_id` in `posts` to avoid probe pagination.
- Scheduled update watches (e.g., auto-update monthly on dependency
  version bumps).
- Full reverse-engineering of Medium/Substack unpublish (no API).
- Multi-author support.
- Delete-and-recreate for Dev.to instead of PUT (if Forem PUT proves
  insufficient for unpublish semantics).

### Known limitations

- Medium / Substack unpublish is manual-only (no API).
- Update is serialized per-slug by the shared lock; parallel updates
  across different posts are fine.
- `site-update` step defaults to PR mode for safety. `direct` mode
  exists via `config.updates.site_update_mode='direct'` for operators
  who opt out of the preview gate — documented, not recommended.
- Research-page regeneration on update does a full re-render even
  if the underlying research doc did not change; relies on byte-level
  git diff to produce a no-op commit when unchanged.
