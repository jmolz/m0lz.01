# Feature: Phase 5 — Three-Reviewer Adversarial Evaluation

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the evaluation phase of the idea-to-distribution pipeline: three independent reviewers (Claude structural, GPT-5.4 high adversarial, GPT-5.4 xhigh methodology) run in parallel, write findings to disk, and the CLI synthesizes their outputs into a consensus/majority/single-reviewer verdict. Includes a deterministic `structural-autocheck` sidecar that pre-populates the structural reviewer's input with machine-found issues (frontmatter schema, placeholder scan, benchmark-claim cross-check, broken-link check, MDX parse). The CLI manages workspace, ingestion, synthesis, and phase gating; the Claude Code skill orchestrates the AI reviewers. This is the fourth pipeline phase, consuming draft MDX from Phase 4 and feeding into Phase 6 (publish).

## User Story

As a content author, I want every draft evaluated by three independent reviewers running in parallel — each with a distinct role (structural, adversarial, methodology) — with their findings automatically synthesized into consensus/majority/single-reviewer categories and a clear pass/fail verdict, so that I have objective multi-model checks and balances before `blog publish` runs, and I can trust that the categorization is reproducible and transparent (not a black-box LLM judgment call).

## Success Criteria

- [ ] `blog evaluate init <slug>` creates `.blog-agent/evaluations/{slug}/`, writes `manifest.json` listing expected reviewers (content-type-aware: methodology skipped for `analysis-opinion`). Idempotent.

## Data Contract: ReviewerOutput

Each reviewer writes a JSON file at `.blog-agent/evaluations/{slug}/{reviewer}.json`:

```typescript
export interface Issue {
  id: string;              // stable: sha256(reviewer + normalizedTitle + normalizedDescription).slice(0, 12)
  category: string;        // e.g., 'frontmatter', 'thesis', 'sample-size', 'broken-link'
  severity: 'low' | 'medium' | 'high';
  title: string;           // short, one-line
  description: string;     // full explanation
  source?: 'autocheck' | 'reviewer';  // structural-only: tags autocheck-found issues
}

export interface ReviewerOutput {
  reviewer: 'structural' | 'adversarial' | 'methodology';
  model: string;           // e.g., 'claude-code', 'gpt-5.4-high', 'gpt-5.4-xhigh'
  passed: boolean;         // reviewer's own pass/fail verdict (advisory; synthesis computes the hard verdict)
  issues: Issue[];
  report_path?: string;    // optional pointer to a human-readable markdown report
}
```

Validation rules: all fields present, types correct, `reviewer` is one of three enum values, `issues` is an array (may be empty), each `Issue` has all required fields.

---

## Synthesis Algorithm (two-tier matching)

1. **Normalize** each issue: lowercase, collapse whitespace, strip punctuation. Produce `normalized = normalize(title) + '\n' + normalize(description)`.
2. **Tier 1 — Exact fingerprint match:** SHA-256 of `normalized`. Issues with identical fingerprints across reviewers are matched directly.
3. **Tier 2 — Jaccard fallback:** For unmatched issues, compute Jaccard token overlap between all pairs. Tokens = unique words after normalization. Match when `|A ∩ B| / |A ∪ B| ≥ JACCARD_THRESHOLD` (default `0.6`).
4. **Union-find cluster:** Group matched issues across reviewers into clusters. Each cluster has a set of reviewer contributions.
5. **Categorize:**
   - **Consensus:** cluster touched by all expected reviewers (3/3 or 2/2 for `analysis-opinion`).
   - **Majority:** cluster touched by a strict majority but not all (2/3; N/A for 2-reviewer mode — skip this bucket when only 2 reviewers).
   - **Single:** cluster touched by exactly one reviewer.
6. **Verdict:** `fail` if `consensus_issues > 0 OR majority_issues > 0`; otherwise `pass`.

`JACCARD_THRESHOLD` is a named constant exported from `synthesize.ts` for tuning and testability.

---

## Autocheck Lints (deterministic)

`runStructuralAutocheck(slug, paths)` produces `Issue[]` sorted by `(category, id)`. Lints:

| Category | Check | Severity |
|----------|-------|----------|
| `frontmatter-schema` | `validateFrontmatter()` fails for any field | high |
| `frontmatter-placeholder` | `title`/`description` still `{{title}}`/`{{description}}` | high |
| `placeholder-section` | `PLACEHOLDER_PATTERN` matches anywhere in MDX body | high |
| `broken-internal-link` | Internal `](/...)` or `](#...)` target not resolvable | medium |
| `mdx-parse` | MDX contains unescaped `<` outside code blocks, unbalanced fence | high |
| `benchmark-claim-unbacked` | Prose claims a numeric value that does not appear in `results.json.data` (numeric token extraction + literal match) | medium |
| `missing-companion-repo` | `has_benchmarks=1` but `companion_repo` missing from frontmatter | medium |

Issues are tagged `"source": "autocheck"`. The Claude structural reviewer merges these with judgment-based findings (reading level, prose quality, source sufficiency) tagged `"source": "reviewer"` to produce the final `structural.json`.

### E2E Validation Steps

1. Advance a post through draft to evaluate (Phase 4 pipeline).
2. `blog evaluate init <slug>` — verify manifest.
3. `blog evaluate structural-autocheck <slug>` — verify `structural.lint.json` created and deterministic on rerun.
4. Manually craft three reviewer JSON files at `.blog-agent/evaluations/{slug}/{structural,adversarial,methodology}.json`.
5. `blog evaluate record <slug> --reviewer structural --report structural.md --issues structural.json` (repeat for each).
6. `blog evaluate show <slug>` — verify all three recorded.
7. `blog evaluate synthesize <slug>` — verify verdict printed, `synthesis.md` written, DB row present.
8. If pass: `blog evaluate complete <slug>` — verify phase = publish.
9. If fail: `blog evaluate reject <slug>` — verify phase = draft.

## Notes

- **No schema migration:** `evaluations` and `evaluation_synthesis` tables already exist in V1 (`SCHEMA_V1_SQL`). `SCHEMA_VERSION` stays at 2.
- **Transparency principle:** All deterministic work (autocheck, synthesis, verdict) lives in the CLI. The skill's AI work is confined to reviewer-specific judgment. Categorization is reproducible from inputs — graders can re-derive it.
- **Why two-tier matching (fingerprint + Jaccard):** Exact fingerprint catches identical findings from reviewers who agree on exact phrasing (fast, deterministic). Jaccard handles paraphrase (different models use different words for the same issue). LLM-assisted dedup was rejected because it makes synthesis non-reproducible and untestable. The hybrid gives best-in-class accuracy while staying fully deterministic.
- **Issue severity:** tracked on each `Issue` but not used for verdict computation in MVP. Future: weight consensus/majority counts by severity (`high` consensus issue = harder fail than `low` consensus).
- **Structural autocheck is a sidecar, not a replacement:** Claude still does structural review with judgment. Autocheck pre-populates `structural.lint.json` with mechanical findings; Claude merges with `"source": "reviewer"`-tagged judgment findings when writing the final `structural.json`. This captures the mechanical checks with 100% reliability without removing human-style judgment.
- **Codex CLI failure mode:** If `codex exec` fails mid-run, the skill documents how to retry just the failed reviewer (no need to re-record the successful ones — `recordReview` is idempotent per reviewer).
- **`analysis-opinion` two-reviewer mode:** No majority bucket — synthesis either has consensus (2/2) or single (1/2) clusters. Verdict logic still correct: `fail` if `consensus > 0 || majority > 0` — with majority always 0 in 2-reviewer mode, the rule reduces to "fail if consensus > 0".
- **Score formula:** `evaluation_score = 1 - (consensus*2 + majority) / max(1, total_issues)`. Provides a 0–1 quality metric for the metrics dashboard. Placeholder — refine after real data is available.
- **Rejecting non-loss of history:** `reject` does not delete prior `evaluations` rows. On re-record after reject, new rows are inserted with `is_update_review=1` (UPSERT via `INSERT OR REPLACE` would clobber history; instead, synthesis always reads the most recent row per reviewer via `ORDER BY run_at DESC LIMIT 1`). This gives full audit trail.
