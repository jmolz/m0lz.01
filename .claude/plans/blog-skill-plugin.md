## Overview

Build a Claude Code plugin containing a single `/blog` orchestration skill plus a new `blog agent` CLI subcommand family that implements the structural safety boundary. The skill captures user intent in natural language, proposes an execution plan, writes it to disk with a SHA256 hash binding approval to exact content, and hands off to `blog agent apply` — the CLI command that validates the hash, verifies workspace root, and executes concrete CLI steps. The CLI is the gate, not the skill prose.

## User Story

As m0lz.01's operator, I want to describe content intent in natural language inside a Claude Code session, so that I never remember CLI syntax. The agent proposes a plan I can review; once approved, the CLI refuses to execute any plan whose content was modified after approval.

**Plan shape (v2 schema, SHA256-bound):**

```json
{
  "schema_version": "2",
  "plan_id": "01JH0K...ULID",
  "slug": "m0lz-01-launch",
  "workspace_root": "/Users/jacobmolz/blog",
  "created_at": "2026-04-19T18:00:00Z",
  "approved_at": null,
  "payload_hash": null,
  "intent": "Launch post for existing npm package jmolz/m0lz.01; Show HN + hub + Dev.to",
  "content_type": "project-launch",
  "depth": "fast-path",
  "venues": ["hub", "devto", "hn"],
  "expected_start_phase": "research",
  "steps": [
    {
      "command": "blog research finalize",
      "args": ["m0lz-01-launch"],
      "checkpoint_message": "Research sources look good — finalize?",
      "preconditions": { "phase": "research", "min_sources": 3 }
    },
    {
      "command": "blog draft init",
      "args": ["m0lz-01-launch"],
      "checkpoint_message": "Ready to scaffold the draft?",
      "preconditions": { "phase": "draft" }
    },
    {
      "command": "blog evaluate record",
      "args": ["m0lz-01-launch", "--reviewer", "structural", "--report", ".blog-agent/evaluations/m0lz-01-launch/structural.md", "--issues", ".blog-agent/evaluations/m0lz-01-launch/structural.json"],
      "checkpoint_message": "Record structural reviewer output?",
      "preconditions": { "phase": "evaluate" }
    }
  ]
}
```

Hash derivation:
- Build canonical JSON: drop `approved_at` + `payload_hash` fields, sort keys recursively, serialize with `JSON.stringify(obj, Object.keys(obj).sort())` (stable-stringify)
- `payload_hash = sha256(canonical).hex()`
- On approval: set `approved_at` to `new Date().toISOString()`, compute hash, set `payload_hash`, write file
- On apply: read file, recompute hash from canonical form, compare to `payload_hash`. Mismatch → exit 2.

**Skill frontmatter:**

```yaml
---
name: blog
description: Orchestrate m0lz.01 content lifecycle. Describe your intent in natural language; I'll propose a plan, get your approval, and hand off to the CLI for structural-safety-gated execution.
argument-hint: [what you want to do]
allowed-tools: Bash(blog:*) Bash(gh:*) Read Write Edit Grep Glob

```

Every protocol step the skill performs is either a `blog` call (covered by `Bash(blog:*)`), a Read/Write/Edit on `.blog-agent/**`, or a Grep/Glob search. No orphan `node -e`, `cat`, or `head` invocations.

### Task 0d: `blog agent` CLI subcommand family

**File:** `src/cli/agent.ts`, edits to `src/cli/index.ts`
**Action:** CREATE + MODIFY
**Description:** Five subcommands registered under `agent`:
1. `blog agent preflight [--json]` — workspace root + config validity + CLI version; emits envelope or human
2. `blog agent plan <slug> --intent <text> [--content-type <t>] [--depth <d>] [--venues <list>]` — writes a plan skeleton with `approved_at: null`, `payload_hash: null`; the skill uses this (NOT hand-edit) so the file shape is always validator-compliant
3. **`blog agent approve <plan-path>`** — atomic: reads plan, recomputes canonical JSON hash, writes `approved_at` + `payload_hash` back in a single fsync. Skill calls this on user approval instead of hand-editing the file.
4. `blog agent verify <plan-path>` — dry-run validator; exit 0 if applyable, exit 2 with `errorCode` in stderr otherwise
5. `blog agent apply <plan-path> [--restart]` — validate, then iterate steps via `src/core/plan-file/apply.ts`. **Default behavior is RESUMABLE** (inherits Phase 6/7 publish-pipeline semantics): reads receipt if present, skips already-completed steps, executes from first non-completed step. `--restart` flag forces re-execution from step 0. Receipt path: `.blog-agent/plans/<slug>.receipt.json`.

**Error code taxonomy** (emitted as `[AGENT_ERROR] <CODE>: <msg>` on stderr; `src/core/plan-file/validator.ts` exports the enum):
- `SCHEMA_INVALID` (exit 2) — JSON schema / enum violation
- `NO_APPROVAL` (exit 2) — `approved_at: null`
- `HASH_MISMATCH` (exit 2) — payload modified after approval
- `WORKSPACE_MISMATCH` (exit 2) — plan's `workspace_root` ≠ detected root
- `UNKNOWN_COMMAND` (exit 2) — step's `command` isn't a registered subcommand
- `STEP_FAILED` (exit 1) — a step's `blog …` exited non-zero (receipt captures which step + exit code + stderr tail)
- `RECEIPT_CONFLICT` (exit 2) — receipt claims completion state that contradicts current `blog status --json` (protects against manually-edited receipt)

**Validate:** `blog agent preflight --json` returns valid envelope; `blog agent approve` on unapproved plan sets both fields atomically; `blog agent verify` on tampered plan exits 2 with `HASH_MISMATCH`; `blog agent apply` resumable via receipt; `--restart` flag re-runs from step 0.

## Notes

### Design decisions (v3)

- **Single `/blog` skill + CLI-native safety boundary** (reversal of v2 two-skill split). The two-skill version moved the boundary into a UX split — but review surfaced that the `/blog-execute` side remained prose-enforced. Moving the boundary into `blog agent apply` (which validates SHA256 hash + `approved_at` + workspace root structurally) means a single skill is sufficient. Simpler UX; stronger enforcement. Terraform's two-command pattern applies when the tool has no runtime enforcement inside the binary; the new `agent` subcommand family IS that runtime enforcement.
- **SHA256 payload hash binds approval to exact content.** Approve-Monday-edit-Tuesday-execute-Wednesday is rejected structurally. Plan becomes an immutable review artifact; receipts are the mutable record.
- **Workspace-root detection is CLI-wide, not skill-only.** Every `.blog-agent/` path construction in `src/cli/*.ts` routes through `workspaceRelative()`. The skill's `blog agent preflight --json` uses the same resolver. No cwd-surprise.
- **Plugin ships via npm tarball.** Three install paths in docs (bundled, repo-clone, contributor symlink); `package.json#files` + `verify-pack.mjs` enforce the distribution contract.
- **Concrete step schema.** Plan steps are `{command, args}` pairs → direct `child_process.spawn('blog', args)`. No abstract actions, no runtime interpretation beyond exit-code propagation.
- **Receipt file separate from plan file.** Plan stays immutable post-approval; receipt accumulates execution state. Clean separation of "what was reviewed" vs "what happened."

### Trade-offs

- **`blog agent` is new public CLI surface** — permanent (5 subcommands: preflight, plan, approve, verify, apply). Versioned plan schema + JSON envelope mitigate evolution pain.
- **Workspace refactor is minimal, not CLI-wide** — Option C (chdir at startup + preserved user-path helper) touches `src/cli/index.ts` + ~15 flag-parser edits across 3 files, not 41 call sites. Existing 730 tests stay green because their fixtures already `mkdtemp` + operate in the test's cwd.
- **`--dry-run` on network-bound steps is a real CLI feature, not test scaffolding** — operators get a preview mode for free; fixture test reuses the same code path as production dry-runs. ~7 step implementations touched.
- **No LLM-behavior integration test** — skill prose can't be mocked meaningfully. Fixture integration tests the mechanics the skill relies on, including resumability + hash-mismatch + workspace-mismatch; live dogfood (Task 11) is the final behavioral sign-off.

### Known limitations / follow-up

- **Plan editing after proposal** — skill can offer "edit step N" before approval; editing after approval invalidates the hash (by design; user re-approves). No in-place amendment flow in v0.2.
- **Marketplace publication** — v0.3 concern
- **`--json` on write commands** — v0.2.1 if parse friction surfaces
- **Multi-tenant plugin** — v0.3

### Revision history

- **v1** — single `/blog` skill, prose-gated approvals, deleted legacy `/skills/*.md`, parsed `blog status` human output. 6 Criticals flagged by parallel Claude + Codex review: status-as-state-sensor, dogfood-only integration, premature deletion, invented content type, CWD-bound workspace, prose-gated approval.
- **v2** — applied three "best-in-class" decisions: `--json` prerequisite, two-skill structural split (`/blog` + `/blog-execute`), plugin+symlink hybrid. Preserved `/skills/*.md`. Fixed enum violation + CWD narrowing + added workspace-root detection.
