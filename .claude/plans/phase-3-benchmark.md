# Feature: Phase 3 — Benchmark and Test Harness

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the benchmark layer of the idea-to-distribution pipeline: a Claude Code skill that scaffolds test harnesses from research targets (AI-heavy) and a typed TypeScript core + CLI bridge that manages benchmark state in SQLite, captures environment snapshots, stores raw results, scaffolds companion repo directory structure, and generates METHODOLOGY.md. This is the second pipeline phase after Phase 2's research and feeds directly into Phase 4 (drafting).

## User Story

As a content author, I want to prompt Claude Code with a finalized research slug and have it scaffold a reproducible test harness from my benchmark targets, capture my environment, run the tests, and store structured results with METHODOLOGY.md — so that every benchmark claim in my post is backed by verifiable primary-source data that readers can reproduce.

## Success Criteria

- [ ] `blog benchmark init <slug>` transitions post from `research` to `benchmark` phase, creates `.blog-agent/benchmarks/<slug>/` directory, and parses benchmark targets from the research doc
- [ ] `blog benchmark init` is content-type-aware: `required` runs normally, `optional` warns but proceeds, `skip` transitions directly to `draft` phase
- [ ] `blog benchmark env <slug>` captures environment snapshot to `.blog-agent/benchmarks/<slug>/environment.json` and records it in the `benchmarks` table
- [ ] `blog benchmark run <slug>` creates/updates a benchmark run row in SQLite with status tracking (pending → running → completed/failed)
- [ ] `blog benchmark show <slug>` prints benchmark state: run count, environment captured, results path, content type requirement
- [ ] `blog benchmark skip <slug>` gracefully skips benchmarks for `analysis-opinion` posts (or any post where config says `skip`), advancing directly to `draft` phase with `has_benchmarks = FALSE`
- [ ] Environment capture includes: OS, architecture, CPU, memory, Node.js version, npm version, and a timestamp
- [ ] METHODOLOGY.md template generated at `.blog-agent/benchmarks/<slug>/METHODOLOGY.md` with environment details and reproduction steps
- [ ] Companion repo structure scaffolded at `.blog-agent/repos/<slug>/` with `src/`, `results/`, `METHODOLOGY.md`, `LICENSE`, and `README.md`
- [ ] Results stored at `.blog-agent/benchmarks/<slug>/results.json` with structured format
- [ ] `skills/blog-benchmark.md` is a complete workflow (replaces stub), referencing all CLI commands and guiding test harness scaffolding
- [ ] All validation commands pass with zero errors
- [ ] No regressions in Phase 1 or Phase 2 commands

### Task 3: Benchmark State Module

**File:** `src/core/benchmark/state.ts`
**Action:** CREATE
**Description:** Post lifecycle helpers for the benchmark phase. Exports:

```typescript
export function getBenchmarkPost(db: Database.Database, slug: string): PostRow | undefined;

export function initBenchmark(
  db: Database.Database,
  slug: string,
  benchmarkDir: string,
  researchDir: string,
): { targets: string[]; benchmarkPath: string };

export function skipBenchmark(
  db: Database.Database,
  slug: string,
): void;

export type BenchmarkRequirement = 'required' | 'optional' | 'skip';

export function getBenchmarkRequirement(
  contentType: ContentType,
  config: BlogConfig,
): BenchmarkRequirement;

export function createBenchmarkRun(
  db: Database.Database,
  slug: string,
  environmentJson: string,
  resultsPath: string,
): number;  // returns benchmark row id

export function updateBenchmarkStatus(
  db: Database.Database,
  runId: number,
  status: BenchmarkStatus,
): void;

export function listBenchmarkRuns(
  db: Database.Database,
  slug: string,
): BenchmarkRow[];

export function completeBenchmark(
  db: Database.Database,
  slug: string,
): void;
```

**Implementation notes:**

- `getBenchmarkPost` mirrors `getResearchPost` pattern — returns `PostRow | undefined`, throws if post exists but is not in `benchmark` phase.
- `initBenchmark`:
  1. Reads the post row — must be in phase `research` (throws otherwise).
  2. Reads the research doc via `readResearchDocument()` to extract `benchmark_targets`.
  3. Parses targets from the `benchmark_targets` field (split on `\n`, filter lines starting with `- `).
  4. Calls `advancePhase(db, slug, 'benchmark')`.
  5. Creates `.blog-agent/benchmarks/<slug>/` directory.
  6. Returns the parsed targets and the benchmark directory path.
- `skipBenchmark`:
  1. Post must be in `research` phase.
  2. Sets `has_benchmarks = FALSE` on the post.
  3. Advances phase directly to `draft` (skipping `benchmark`).
- `getBenchmarkRequirement` reads `config.content_types[contentType].benchmark` and returns it as typed string.
- `createBenchmarkRun` inserts into `benchmarks` table with status `'pending'`, returns the id.
- `updateBenchmarkStatus` updates status + timestamps (`started_at` when `'running'`, `completed_at` when `'completed'`/`'failed'`).
- `completeBenchmark` sets `has_benchmarks = TRUE` on the post and advances phase to `draft`.

**Depends on:** none (uses existing `advancePhase`, `readResearchDocument`)
**Validate:** Unit tests with `:memory:` DB — init transitions to benchmark, skip transitions to draft, run lifecycle (pending → running → completed), getBenchmarkPost throws for non-benchmark posts.

### Task 6: CLI Subcommand Group

**File:** `src/cli/benchmark.ts`
**Action:** CREATE
**Description:** Commander subcommand group for `blog benchmark`. Mirrors the handler pattern in `src/cli/research.ts`.

**Exports:**

```typescript
export interface BenchmarkPaths {
  dbPath?: string;
  benchmarkDir?: string;
  reposDir?: string;
  researchDir?: string;
  configPath?: string;
}

export function runBenchmarkInit(slug: string, paths?: BenchmarkPaths): void;
export function runBenchmarkEnv(slug: string, paths?: BenchmarkPaths): void;
export function runBenchmarkRun(
  slug: string,
  opts: { resultsFile?: string },
  paths?: BenchmarkPaths,
): void;
export function runBenchmarkShow(slug: string, paths?: BenchmarkPaths): void;
export function runBenchmarkSkip(slug: string, paths?: BenchmarkPaths): void;
export function registerBenchmark(program: Command): void;
```

**`runBenchmarkInit` flow:**

1. Validate slug at CLI boundary.
2. Load config. Check `getBenchmarkRequirement(post.content_type, config)`.
3. If `skip`: print "Content type `<type>` does not require benchmarks. Use `blog benchmark skip <slug>` to advance to draft." and `process.exitCode = 1; return`.
4. If `optional`: print warning "Benchmarks are optional for `<type>` content. Proceeding."
5. Open DB. Call `initBenchmark(db, slug, benchmarkDir, researchDir)`.
6. Print parsed benchmark targets.
7. Print "Benchmark workspace created at `.blog-agent/benchmarks/<slug>/`".

**`runBenchmarkEnv` flow:**

1. Validate slug. Open DB.
2. Verify post is in `benchmark` phase via `getBenchmarkPost`.
3. Call `captureEnvironment()`.
4. Call `writeEnvironment(benchmarkDir, slug, env)`.
5. Print formatted environment summary.

**`runBenchmarkRun` flow:**

1. Validate slug. Open DB.
2. Verify post is in `benchmark` phase.
3. Check that environment.json exists (error if not — `run blog benchmark env <slug> first`).
4. Read environment JSON. Create a benchmark run row (`createBenchmarkRun`).
5. Update status to `running`.
6. If `--results-file` provided, read the file as JSON and store via `writeResults`. Update status to `completed`.
7. If no results file, print "Run row created (id=N, status=running). Provide results with `blog benchmark run <slug> --results-file path/to/results.json`."
8. Print run summary.

**`runBenchmarkShow` flow:**

1. Validate slug. Open DB.
2. Get post (accept both `research` and `benchmark` phases — show can be called pre-init).
3. Print: slug, phase, content_type, benchmark requirement, run count, environment captured (yes/no), results path.

**`runBenchmarkSkip` flow:**

1. Validate slug. Open DB. Load config.
2. Check requirement. If `required`: print "Benchmarks are required for `<type>`. Cannot skip." and `process.exitCode = 1`.
3. Call `skipBenchmark(db, slug)`.
4. Print "Skipped benchmarks for <slug>. Phase advanced to draft."

**Commander wiring:** nested subcommands, mirrors `research.ts` pattern.

**Depends on:** Tasks 2, 3, 4, 5
**Validate:** All five CLI handlers covered in `tests/benchmark-cli.test.ts` with happy + error paths.

## Testing Strategy

### Unit Tests

- **Where:** `tests/*.test.ts` (flat structure — matches Phase 1/2)
- **Pattern:** Vitest, `:memory:` DBs, `mkdtempSync` for filesystem, mock `console.log`/`console.error` via `vi.spyOn`
- **Minimum:** 1 happy + 1 edge + 1 error per module

### Integration Tests

- CLI handlers imported directly and invoked with injected paths (the `tests/research-cli.test.ts` pattern)
- Phase transition tested end-to-end: research → benchmark → draft via both the normal and skip paths

1. `npm run build`

## Notes

- **No schema migration needed.** The `benchmarks` table was created in the V1 schema (Phase 1). All columns needed for Phase 3 already exist. `SCHEMA_VERSION` stays at 2.
- **`initBenchmark` owns the research→benchmark phase transition.** This mirrors Phase 2's design where `finalize` is a read-only gate and the next phase's `init` owns the transition. This keeps the contract unambiguous: you can re-run `finalize` safely, and you can only enter `benchmark` via `initBenchmark`.
- **The CLI does not execute tests.** The skill (`/blog-benchmark`) uses Claude Code to write and execute test harnesses. The CLI's `blog benchmark run` only records runs and stores results. This preserves the dual-layer separation: AI-heavy work in skills, mechanical persistence in CLI.
- **`blog benchmark run --results-file`** is the bridge between skill execution and CLI persistence. The skill runs tests, writes `results.json`, then calls the CLI to register it. This avoids the CLI needing to know about arbitrary test frameworks.
- **Content-type routing uses config, not hardcoded values.** `getBenchmarkRequirement` reads from `.blogrc.yaml`'s `content_types[type].benchmark` field, making the behavior configurable.
- **Follow-up (Phase 4 preview):** Drafting phase will read `results.json` for benchmark data tables, `environment.json` for the "Tested on {env}" methodology reference, and the companion repo for code citations. The `BenchmarkResults` type from this phase becomes Phase 4's input contract.
- **Companion repo scaffolding is local only.** GitHub push happens in Phase 6 (publish pipeline). Phase 3 creates the directory structure and initial files; the skill populates `src/` with actual test code.
- **Multiple benchmark runs** are tracked as separate rows in the `benchmarks` table, each with its own `environment_json` and `results_path`. This supports the PRD's requirement for statistical validity across runs and preserving contradictory data.
