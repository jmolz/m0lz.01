# Feature: Phase 4 — Drafting and Visuals

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the drafting layer of the idea-to-distribution pipeline: typed core modules that manage draft state in SQLite, enforce PostFrontmatter schema compliance, format benchmark data for inclusion in MDX, track generated assets (Excalidraw SVGs, charts), read existing site tags for taxonomy consistency, and provide CLI commands for the full draft lifecycle. The `/blog-draft` Claude Code skill is rewritten from its stub to a complete workflow that uses these CLI commands. This is the third pipeline phase, consuming research documents and benchmark data from Phases 2-3, and feeding into Phase 5 (evaluation).

## User Story

As a content author, I want to run `/blog-draft` on a slug with completed research and benchmarks, and have the agent produce a valid MDX post with auto-populated frontmatter, integrated benchmark data tables, asset tracking for diagrams, and tags consistent with my site's existing taxonomy — so that the draft is schema-compliant and ready for adversarial evaluation without manual frontmatter wrangling.

## Success Criteria

- [ ] `blog draft init <slug>` creates draft workspace at `.blog-agent/drafts/{slug}/`, generates template MDX with auto-populated PostFrontmatter, and creates `assets/` subdirectory
- [ ] `blog draft init` is content-type-aware: `technical-deep-dive` includes benchmark data sections, `project-launch` links existing repo, `analysis-opinion` omits benchmark sections
- [ ] `blog draft show <slug>` prints draft state: frontmatter validity, section completeness, asset count, benchmark data availability
- [ ] `blog draft validate <slug>` validates PostFrontmatter schema (all required fields present and typed correctly), checks MDX sections are not placeholders, verifies asset files exist for registered assets
- [ ] `blog draft add-asset <slug>` registers an asset file in the `assets` table with type (excalidraw, chart, image, benchmark_viz) and verifies file exists
- [ ] `blog draft complete <slug>` validates the draft and advances phase to `evaluate`
- [ ] PostFrontmatter interface matches m0lz.00 contract exactly: `title`, `description`, `date`, `tags`, `published`, `canonical?`, `companion_repo?`, `project?`, `medium_url?`, `devto_url?`

- [ ] Asset management: assets tracked in existing `assets` table, asset directory at `.blog-agent/drafts/{slug}/assets/`
- [ ] `skills/blog-draft.md` is a complete workflow (replaces stub), referencing all CLI commands and guiding MDX content generation with benchmark data integration and Excalidraw diagram creation

- [ ] No regressions in Phase 1, Phase 2, or Phase 3 commands

// Phase boundary in library function (from benchmark/state.ts)
export function getDraftPost(db: Database.Database, slug: string): PostRow | undefined {
  const post = db.prepare('SELECT * FROM posts WHERE slug = ?').get(slug) as PostRow | undefined;
  if (post && post.phase !== 'draft') {
    throw new Error(
      `Post '${slug}' is in phase '${post.phase}', not 'draft'. ` +
      `Draft commands only operate on posts in the draft phase.`,
    );
  }
  return post;
}

// CLI boundary (from cli/benchmark.ts)
try {

} catch (e) {
  console.error((e as Error).message);
  process.exitCode = 1;
  return;
}

### Task 1: PostFrontmatter Interface and Validation

**File:** `src/core/draft/frontmatter.ts`
**Action:** CREATE
**Description:** Define the `PostFrontmatter` interface matching the m0lz.00 contract exactly. Implement:
- `generateFrontmatter(post: PostRow, config: BlogConfig)` — auto-populate canonical URL (`config.site.base_url + '/writing/' + post.slug`), companion_repo URL (`'https://github.com/' + config.author.github + '/' + post.slug` when `post.has_benchmarks`), project from `post.project_id`, date from current date, `published: false`, tags as empty array (filled by skill later).
- `validateFrontmatter(fm: PostFrontmatter)` — check required fields (`title`, `description`, `date`, `tags`, `published`) are present and correctly typed. Return `{ ok: boolean; errors: string[] }`.
- `serializeFrontmatter(fm: PostFrontmatter)` — serialize to YAML frontmatter string (delimited by `---`).
- `parseFrontmatter(mdxContent: string)` — parse YAML frontmatter from an MDX string, return `PostFrontmatter`. Use `js-yaml`.
**Depends on:** none
**Validate:** `npm run lint`

### Task 2: Draft State Management

**File:** `src/core/draft/state.ts`
**Action:** CREATE
**Description:** Implement draft phase state management following `src/core/benchmark/state.ts` pattern:
- `getDraftPost(db, slug)` — fetch post with phase enforcement (must be `draft`). Throw if wrong phase.
- `initDraft(db, slug, draftsDir, benchmarkDir, researchDir, config)` — verify post is in `draft` phase, create `.blog-agent/drafts/{slug}/` and `.blog-agent/drafts/{slug}/assets/` directories, generate template MDX via `renderDraftTemplate()` (Task 4), write to `.blog-agent/drafts/{slug}/index.mdx`. Return `{ draftPath: string; frontmatter: PostFrontmatter }`. Idempotent: if draft file already exists, do not overwrite (return existing path).
- `completeDraft(db, slug, draftsDir)` — verify post is in `draft` phase, validate the draft MDX (frontmatter + sections), advance phase to `evaluate` via `advancePhase()`. Throw if validation fails.
- `registerAsset(db, slug, assetType: AssetType, filename: string)` — insert into `assets` table. Idempotent (INSERT OR IGNORE on slug+filename).
- `listAssets(db, slug)` — query assets for a slug, return `AssetRow[]`.
- `draftPath(draftsDir, slug)` — return path to `index.mdx`.
**Depends on:** Task 1
**Validate:** `npm run lint`

### Task 3: Benchmark Data Formatting

**File:** `src/core/draft/benchmark-data.ts`
**Action:** CREATE
**Description:** Format benchmark results and environment for draft inclusion:
- `formatBenchmarkTable(results: BenchmarkResults)` — convert `results.data` (Record<string, unknown>) into a markdown table. Handle nested objects by flattening one level. If data has array values, render each as a row. Return markdown string.
- `formatMethodologyRef(env: EnvironmentSnapshot, slug: string)` — return string like: `Tested on {env.os} {env.arch} ({env.cpus}) with Node.js {env.node_version} -- see [METHODOLOGY.md](https://github.com/jmolz/{slug}/blob/main/METHODOLOGY.md) for full reproduction steps.`
- `getBenchmarkContext(benchmarkDir: string, slug: string)` — read both `results.json` and `environment.json`, return `{ results: BenchmarkResults | null; environment: EnvironmentSnapshot | null; table: string; methodologyRef: string }`. Gracefully handle missing files.
**Depends on:** none
**Validate:** `npm run lint`

## Notes

- **No schema migration:** The `assets` table already exists in V1 (`SCHEMA_V1_SQL`). Draft phase uses it directly. `SCHEMA_VERSION` stays at 2.
- **Excalidraw integration:** The CLI tracks assets in the DB and manages the `assets/` directory. Actual SVG generation is the skill's responsibility using Claude Code's `excalidraw-diagram` skill. The CLI does not invoke Excalidraw.
- **Benchmark data visualization:** The CLI formats data as markdown tables. Chart/graph generation (if any) is the skill's domain. The `benchmark_viz` asset type exists for tracking these.
- **Draft file location:** `.blog-agent/drafts/{slug}/index.mdx` (not `.blog-agent/drafts/{slug}.mdx`). Using a directory allows co-locating assets. This differs slightly from the PRD's `{slug}.mdx` — the directory approach is better for asset management.
- **Tag taxonomy is best-effort:** If m0lz.00 is not at the configured path, `readExistingTags()` returns an empty array. Tags are ultimately chosen by the skill (Claude Code), not enforced by the CLI.
- **Phase transition timing:** The post enters `draft` phase during `benchmark complete` or `benchmark skip` (Phase 3). `draft init` does NOT change phase — it sets up the workspace. `draft complete` advances to `evaluate`.
- **Content-type routing in templates:** Rather than a single template file with conditionals, `renderDraftTemplate()` programmatically builds the MDX structure based on content type. This is cleaner than template branching.
- **Frontmatter `title` and `description`:** Auto-generated as placeholders (`"{{title}}"`, `"{{description}}"`) since the skill writes the actual values. `validateFrontmatter()` rejects these placeholders.
