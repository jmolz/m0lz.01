# Feature: Phase 2 — Research Pipeline

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the research layer of the idea-to-distribution pipeline: a Claude Code skill that drives AI-heavy research (web search, source gathering, benchmark target identification, thesis drafting) together with a typed TypeScript core + CLI bridge that persists research state to SQLite and writes structured research documents to `.blog-agent/research/<slug>.md`. This is the first pipeline phase after Phase 1's foundation and feeds directly into Phase 3 (benchmark).

## User Story

As a content author, I want to prompt Claude Code with a topic (vague or detailed) and have it produce a validated, source-backed research document with benchmark targets — while a CLI safely tracks state in the blog-agent SQLite — so that the research phase is both AI-driven and mechanically auditable.

## Success Criteria

- [ ] Schema migration v1 to v2 adds `UNIQUE(post_slug, url)` index on `sources`; idempotent on re-open
- [ ] `blog research init <slug>` creates a post row in phase `research` and writes `.blog-agent/research/<slug>.md` from a template
- [ ] `blog research init` is idempotent and refuses to overwrite an existing doc without `--force`
- [ ] `blog research add-source <slug> --url <u>` persists a source and dedupes by `(post_slug, url)`
- [ ] `blog research add-source` errors clearly when the slug does not exist
- [ ] `blog research show <slug>` prints phase, source count, content type, document path
- [ ] `blog research finalize <slug>` enforces config `evaluation.min_sources` and required-section checks
- [ ] Research document round-trips (write then read preserves all sections)
- [ ] `skills/blog-research.md` is a complete workflow (directed + exploratory), replacing the stub
- [ ] All validation commands pass with zero errors
- [ ] No regressions in Phase 1 commands (`init --import`, `status`, `metrics`, `ideas`)

### Task 6: CLI Subcommand Group

**File:** `src/cli/research.ts`
**Action:** CREATE
**Description:** Commander subcommand group for `blog research`. Mirrors the handler pattern in `src/cli/ideas.ts`.

**Exports:**

```
export function runResearchInit(
  slug: string,
  opts: { topic: string; mode?: Mode; contentType?: ContentType; force?: boolean },
  paths?: { dbPath?: string; researchDir?: string; configPath?: string },
): void;

export function runResearchAddSource(
  slug: string,
  opts: { url: string; title?: string; excerpt?: string; type?: SourceType },
  paths?: { dbPath?: string },
): void;

export function runResearchShow(
  slug: string,
  paths?: { dbPath?: string; researchDir?: string },
): void;

export function runResearchFinalize(
  slug: string,
  paths?: { dbPath?: string; researchDir?: string; configPath?: string },
): void;

export function registerResearch(program: Command): void;
```

**`runResearchInit` flow:**

1. If `contentType` not provided, call `detectContentType(topic)` from Phase 1's `content-types.ts`.
2. Default `mode` to `'exploratory'` if not passed.
3. Open DB (with `finally` close). Call `initResearchPost` — print "Created research entry for <slug>" or "Entry already exists — continuing".
4. Build `ResearchDocument` with empty bodies, ISO `created_at`, slug/topic/mode/content_type populated.
5. Call `writeResearchDocument` with `force: opts.force ?? false`. If doc already exists without `--force`: print a descriptive error and `process.exitCode = 1`.
6. Print final path on success.

**`runResearchAddSource` flow:**

1. Open DB. Call `addSource(db, slug, url, {title, excerpt, sourceType: opts.type})`.
2. Print `Added source (id=N)` or `Source already tracked (id=N)`.
3. Errors (missing post) caught and printed to stderr with `process.exitCode = 1`.

**`runResearchShow` flow:**

1. Open DB. Fetch post via `getResearchPost`. If missing: print error, exit 1.
2. Print: slug, phase, mode, content_type, source count, doc path, doc validation summary (ok / N missing / N empty).

**`runResearchFinalize` flow:**

1. Load config via `loadConfig` (from `paths.configPath ?? CONFIG_PATH`). Read `evaluation.min_sources`.
2. Open DB. Verify post exists.
3. `countSources` — if count < min, print `Only N sources (min M required)` to stderr, exit 1.
4. Run `validateResearchDocument` on the doc file — if any required section missing or empty, print the list to stderr, exit 1.
5. On success: print `Research phase ready for <slug>. N sources, doc validated.` Do NOT transition phase here — Phase 3 (benchmark) owns that transition; finalize is a read-only gate.

**Commander wiring:** nested subcommands, mirrors `ideas.ts` pattern. Options documented in Commander help text.

**Depends on:** Tasks 3, 4, 5
**Validate:** All four CLI handlers covered in `tests/research-cli.test.ts` with happy + error paths.

**Required content sections:**

   - Run `init`, gather an initial 3 to 5 sources with WebSearch
   - **Checkpoint 1 — Scope confirm:** present sources and proposed angles to user; wait for direction before deeper research

   - **Checkpoint 2 — Pre-finalize:** present thesis + benchmark targets to user; wait for approval
   - Run `finalize`
5. **Benchmark target guidance** — the "Benchmark Targets" section must list specific, testable claims (format: `- {claim}: {how it would be tested}`) so Phase 3 can pick them up mechanically
6. **Source excerpt guidance** — every `add-source --excerpt` call must state *why* the source matters (claim supported, data point provided), not just summarize the page
7. **Output contract** — reference the sections downstream phases depend on (thesis feeds draft opening; benchmark targets feed benchmark phase; sources feed evaluation structural review)

## Notes

- **Finalize is a gate, not a phase transition.** Phase 3 (benchmark) will own the research to benchmark transition via `advancePhase`. Keeping finalize read-only means re-running it is safe and the contract is unambiguous.
- **Source excerpt is the UX lever.** Instead of numeric source ranking, every source carries a short excerpt naming *why it matters* (claim supported, datum provided). This surfaces signal for downstream consumers (draft, evaluation) without a separate ranking mechanism. The skill enforces this in its workflow prompts.
- **Exploratory checkpoints are minimal but effective.** Two pauses — scope confirm after initial sweep, pre-finalize before gating — balance efficiency against user control. More checkpoints add friction; fewer risks divergence from author intent.
- **Migration test deliberately skips the "pre-existing duplicates" edge case.** V1 databases in the wild (Jacob's workspace) have no sources at all because Phase 1 only imported posts, not sources. If v1 users had populated sources with duplicates, the unique index creation would fail — but that case doesn't exist. A dedicated cleanup script is follow-up work if this assumption is ever wrong.
- **Web search tooling is Claude Code's responsibility.** The skill uses the built-in WebSearch tool. We don't wrap or reimplement it in TypeScript. The CLI is the persistence bridge, not a search client.
- **Content type detection reuse.** Phase 1 already built `detectContentType` in `src/core/draft/content-types.ts`. `runResearchInit` calls it directly; no duplication.
- **Follow-up (Phase 3 preview):** Benchmark phase will need `advancePhase(db, slug, 'benchmark')` and a `blog benchmark init <slug>` that reads the research doc's "Benchmark Targets" section. The `ResearchDocument` type from this phase becomes Phase 3's input contract.
