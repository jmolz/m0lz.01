# Feature: Phase 6 — Publish Pipeline with Resume

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the publish phase of the idea-to-distribution pipeline: an 11-step
sequential pipeline tracked by `pipeline_steps` in SQLite. Each step is
idempotent, checkpointed, and serialized by a filesystem lock per slug. The
pipeline copies MDX + research page to the site repo in a single PR, cross-posts
to Dev.to, generates paste-ready content for Medium and Substack, creates and
pushes companion repos, updates frontmatter with platform URLs (direct push to
main), updates project READMEs, and produces social distribution text. A manual
gate at step 4 ensures nothing reaches production without explicit author
approval. On failure, `blog publish start <slug>` resumes from the first
incomplete step.

## User Story

As a content author, I want to run `blog publish start <slug>` and have the
pipeline handle everything from site PR creation through cross-posting and
social text generation, resuming from where it left off if anything fails, so
that publishing is a single command instead of a 10-step manual process.

## Success Criteria

- [ ] `blog publish start <slug>` initializes the publish phase, creates all 11 pipeline steps as `pending`, and runs them sequentially
- [ ] Pipeline resumes from the first incomplete step on re-run (no duplicate work)
- [ ] Filesystem lock prevents concurrent `blog publish start` invocations on the same slug
- [ ] Step 1 (verify) hard-fails if evaluation not passed
- [ ] Step 2 (research-page) generates research page locally; skips for analysis-opinion without artifacts
- [ ] Step 3 (site-pr) copies MDX + assets + research page to m0lz.00, creates branch and PR via `gh` CLI
- [ ] Step 4 (preview-gate) pauses pipeline if PR not merged, continues on re-run when merged
- [ ] Step 5 (crosspost-devto) creates Dev.to draft article via Forem API with canonical_url; skips if no DEVTO_API_KEY
- [ ] Steps 6-7 generate paste-ready markdown files for Medium and Substack
- [ ] Step 8 probes for companion repo via `gh repo view`, creates via `gh repo create --source=. --push` if missing
- [ ] Step 9 updates frontmatter with platform URLs; direct push to main of site repo
- [ ] Step 10 updates project README; direct push to main of project repo; skips if no `projects[id]` entry in config
- [ ] Step 11 generates LinkedIn + HN social text with content-type-aware formatting
- [ ] `blog publish show <slug>` displays pipeline step status table
- [ ] MDX-to-Markdown converter correctly strips JSX, resolves asset URLs, preserves code blocks
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing functionality (388 tests / 27 suites)

export function getPublishPost(db: Database.Database, slug: string): PostRow {
  const post = db.prepare('SELECT * FROM posts WHERE slug = ?').get(slug) as PostRow | undefined;

### Task 3: Publish Lock

**File:** `src/core/publish/lock.ts`
**Action:** CREATE
**Description:** Copy `acquireEvaluateLock` pattern from `src/core/evaluate/state.ts` (lines ~95-135). Rename to `acquirePublishLock`. Path: `.blog-agent/publish/{slug}/.publish.lock`. Behavior:
1. Ensure workspace directory exists (`mkdirSync recursive`)
2. Attempt atomic `openSync(lockPath, 'wx')` (O_CREAT | O_EXCL)
3. Write PID stamp: `writeFileSync(fd, String(process.pid))`
4. On EEXIST: read PID from existing lock, check if process alive via `process.kill(pid, 0)`. If dead, reclaim by unlinking and retry.
5. Spin-wait with `Atomics.wait(new Int32Array(new SharedArrayBuffer(4)), 0, 0, 50)` if lock is held by live process, up to timeout
6. Return release function: `() => { unlinkSync(lockPath); }`

**Depends on:** none
**Validate:** `npx tsc --noEmit`

### Task 5: Pipeline Steps CRUD

**File:** `src/core/publish/steps-crud.ts`
**Action:** CREATE
**Description:** CRUD operations for `pipeline_steps` table:
- `createPipelineSteps(db, slug, contentType, config)` — inserts 11 rows via `INSERT OR IGNORE` (idempotent). Uses content-type + config to set initial status: most start `pending`, but steps disabled by config or content-type routing start `skipped` directly. Step number matches array index (1-11).
- `getNextPendingStep(db, slug)` — returns first row where `status IN ('pending', 'failed')` ordered by `step_number`. Null if none.
- `markStepRunning(db, slug, stepName)` — UPDATE status = 'running', started_at = CURRENT_TIMESTAMP
- `markStepCompleted(db, slug, stepName)` — UPDATE status = 'completed', completed_at = CURRENT_TIMESTAMP, error_message = NULL
- `markStepFailed(db, slug, stepName, errorMessage)` — UPDATE status = 'failed', completed_at = CURRENT_TIMESTAMP, error_message
- `markStepSkipped(db, slug, stepName, reason?)` — UPDATE status = 'skipped', completed_at = CURRENT_TIMESTAMP, error_message = reason
- `getPipelineSteps(db, slug)` — SELECT all ordered by step_number
- `allStepsComplete(db, slug)` — returns true if COUNT(*) WHERE status IN ('completed', 'skipped') equals COUNT(*)

All SQL uses parameterized `?` placeholders.

**Description:** Pure conversion function `mdxToMarkdown(mdxContent, slug, baseUrl)`:

2. Remove `import` statement lines
3. Process line-by-line with fence-tracking state:

5. URL resolution: replace `./assets/xxx` and `assets/xxx` in image references and links with `{baseUrl}/writing/{slug}/assets/xxx`
6. Collapse 3+ consecutive blank lines to 2

### Task 8: Site PR (Step 3) + Preview Gate (Step 4)

**File:** `src/core/publish/site.ts`
**Action:** CREATE
**Description:**

**`createSitePR(slug, config, paths, db)`** — Step 3:
1. Resolve site repo path: `resolve(dirname(configPath), config.site.repo_path)`
2. Compute repo owner/name from git remote: `execFileSync('git', ['-C', siteRepoPath, 'config', '--get', 'remote.origin.url'])`, parse `owner/name`
3. Branch name: `post/{slug}`
4. Check if branch exists: `git branch --list post/{slug}`. If exists, checkout; else `git checkout -b post/{slug} main`
5. Copy draft MDX: `cpSync(draftPath, join(siteRepoPath, config.site.content_dir, slug, 'index.mdx'))`
6. Copy assets if exist: `cpSync(join(draftsDir, slug, 'assets'), join(..., 'assets'), { recursive: true })`
7. Copy research page if exists at `{researchPagesDir}/{slug}/index.mdx` to `join(siteRepoPath, config.site.research_dir, slug, 'index.mdx')`
8. `git add .`, `git commit -m "feat(post): {slug}"` (skip if no changes — use `git diff --cached --quiet` check first)
9. `git push -u origin post/{slug}`
10. Check for existing PR: `gh pr list --head post/{slug} --json number`. If exists, skip create. Else `gh pr create --title "..." --body "..."`.
11. Write PR number to `{publishDir}/{slug}/pr-number.txt`
12. Return `{ prNumber, prUrl, branchName }`

**`checkPreviewGate(slug, config, paths)`** — Step 4:
1. Read PR number from `{publishDir}/{slug}/pr-number.txt`
2. `gh pr view {prNumber} --repo {owner/name} --json state,mergedAt`
3. Parse JSON response
4. If `state === 'MERGED'`: return `{ merged: true }`
5. Else: return `{ merged: false, message: 'PR #{n} is open — review the Vercel preview, merge when ready, then re-run blog publish start <slug>' }`

All subprocess calls via `execFileSync` with argument arrays.

**Depends on:** Tasks 1, 2

### Task 17: Pipeline Runner

**File:** `src/core/publish/pipeline-runner.ts`
**Action:** CREATE
**Description:** `runPipeline(ctx: PipelineContext)`:
1. Acquire publish lock (from Task 3)
2. Loop:
   a. `getNextPendingStep(db, slug)` — if null, break
   b. Find step def in `PIPELINE_STEPS` by name
   c. `markStepRunning()`
   d. Try: `result = stepDef.execute(ctx)`
   e. Merge `result.urlUpdates` into `ctx.urls`
   f. Translate outcome:
      - `completed` → `markStepCompleted()`, print message, continue
      - `skipped` → `markStepSkipped(reason)`, print message, continue
      - `paused` → leave as `pending`, print message, break
      - `failed` → `markStepFailed(message)`, print error, break
   g. On exception: `markStepFailed(error.message)`, print, break
3. After loop: if `allStepsComplete()`, call `completePublish(db, slug, ctx.urls, publishDir)`
4. Release lock in finally
5. Print summary line: "Pipeline {complete|paused|failed}: {n}/11 steps done"

## Notes

**Design decisions:**
- No manifest/cycle pattern — sequential pipeline, simpler than evaluate's parallel reviewers. `pipeline_steps` table + file lock is sufficient.
- Split state into phase/steps-crud/lock to keep each file focused.
- Split pipeline into types/registry/runner for same reason.
- Research page generated at step 2 (before site PR) so it ships in same atomic review.
- Frontmatter + README updates direct-push to main (mechanical URL additions don't warrant PR review).
- `gh repo create --source=. --push` is idempotent when preceded by `gh repo view` probe.
- File lock copied verbatim from evaluate phase (proven in production).
- `blog publish start <slug>` (not bare `<slug>`) avoids Commander ambiguity with `show` subcommand.

**Known limitations:**
- Asset URLs in cross-posts won't resolve until site PR is merged and deployed. Step ordering (PR + gate before cross-post) mitigates.
- Dev.to tag mapping is best-effort — Dev.to server-side validates and silently drops invalid tags.
- Substack has no publishing API; paste-ready is the only option (documented).
- Medium API is deprecated since March 2023; paste-ready is the only option (documented).
