# Feature: Phase 1 — Foundation

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Scaffold the m0lz.01 project from zero: TypeScript project setup, Commander.js CLI with working `blog init`, `blog status`, `blog metrics`, and `blog ideas` commands, SQLite state database with full schema, config system, existing post import from m0lz.00, and Claude Code skill file stubs. This is the foundation that every subsequent phase builds on.

## User Story

As a content author, I want a working CLI skeleton with config, state management, editorial backlog, and my existing posts imported, so that I have a functional foundation for the research-to-publish pipeline.

## Success Criteria

- [ ] `npm run build` compiles cleanly with zero TypeScript errors
- [ ] `npx blog init` creates `.blog-agent/` directory, `.blog-agent/state.db` with full schema, and prompts for `.blogrc.yaml` config

### Task 4: Database Connection and Migration

**File:** `src/core/db/database.ts`
**Action:** CREATE
**Description:** Database module that opens/creates the SQLite file, enables WAL mode and foreign keys, runs schema migrations using the `user_version` pragma.

Exports:
- `getDatabase(dbPath: string): Database.Database` — opens DB, enables WAL, enables foreign keys, runs migrations if needed
- `closeDatabase(db: Database.Database): void` — clean close with signal handlers

Migration logic: reads `PRAGMA user_version`, compares to `SCHEMA_VERSION`. If version is 0 (new DB), runs full schema creation in a transaction. Future migrations will be `if (currentVersion < 2) { ... }` blocks.

**Depends on:** Task 3
**Validate:** Unit test — create in-memory DB, verify all tables exist, verify user_version matches SCHEMA_VERSION

### Task 7: Post Import Module

**File:** `src/core/migrate/import-posts.ts`
**Action:** CREATE
**Description:** Module that scans a m0lz.00 site repo and imports existing posts into the state database.

Exports:
- `importPosts(db: Database.Database, siteRepoPath: string, baseUrl: string): number` — returns count of imported posts

Logic:
1. Read directories in `{siteRepoPath}/content/posts/`
2. For each directory, read `index.mdx`
3. Parse YAML frontmatter (split on `---`, take second segment, parse with js-yaml)
4. Use `INSERT OR IGNORE INTO posts` (idempotent — running twice does not create duplicates)
5. Map frontmatter fields: slug from directory name, phase='published', mode='imported', site_url=`{baseUrl}/writing/{slug}`, all other fields from frontmatter

**Depends on:** Task 4
**Validate:** Unit test with fixture directory — creates correct rows, idempotent on re-run

### Task 8: CLI Entry Point and Init Command

**File:** `src/cli/index.ts`, `src/cli/init.ts`
**Action:** CREATE
**Description:** Main CLI entry point using Commander.js, and the `blog init` command.

**index.ts** — shebang line (`#!/usr/bin/env node`), creates Commander program, adds all commands, calls `program.parse()`.

**init.ts** — `blog init` command:
1. Create `.blog-agent/` directory and subdirectories: `research/`, `benchmarks/`, `drafts/`, `repos/`, `social/`, `evaluations/`, `research-pages/`
2. Create `.blog-agent/state.db` with full schema (via `getDatabase()`)
3. If `.blogrc.yaml` doesn't exist, copy from `.blogrc.example.yaml` and print message to edit
4. If `.env` doesn't exist, copy from `.env.example` and print message to fill in keys
5. Print success message listing what was created

**`--import` flag:**
When passed, after init:
1. Load config to get `site.repo_path`
2. Call `importPosts(db, resolvedRepoPath, config.site.base_url)`
3. Print summary: "Imported N posts from m0lz.00"

**Depends on:** Task 4, Task 5, Task 7
**Validate:** `npm run build && node dist/cli/index.js init` creates directory structure and DB. With `--import`, imports 5 posts.

## Notes

- **ESM modules**: The project uses `"type": "module"` in package.json. All internal imports must use `.js` extension even for `.ts` source files. This is the TypeScript ESM convention with Node16 module resolution.
- **SQLite synchronous API**: better-sqlite3 is synchronous. Do not introduce async complexity for the database layer. Commander.js action handlers can be sync.
- **Import idempotency**: Use `INSERT OR IGNORE INTO posts` (slug is PRIMARY KEY). Running import twice must not create duplicates and must not throw.
- **No interactive prompts**: The CLI uses Commander.js options/arguments only, not readline prompts. This keeps commands scriptable and testable.
- **Frontmatter parsing**: m0lz.00 posts use YAML between `---` markers. Parse by splitting file content on `/^---$/m`, taking the second segment (index 1), and running through `yaml.load()`. Do not use a heavy MDX parser.
- **Config path resolution**: Paths in `.blogrc.yaml` (like `site.repo_path: "../m0lz.00"`) must be resolved relative to the config file's directory, not CWD. Use `path.resolve(path.dirname(configPath), rawPath)`.
- **No external table library**: Format CLI table output with simple string padding (`String.padEnd()`). Keep dependencies minimal.
